## 设计原则

1. **数据优先于逻辑** — 先定义好表结构，再写业务代码
2. **所有变更通过 Proposal** — 作者资产表的设计必须支持 MOU 流程
3. **向量化是数据的一部分** — 不是外挂系统，是表的固有字段
4. **版本控制思维** — 关键表支持 history/audit trail

## 需要讨论的问题

### Q1: 通用实体表 vs 专用表

方案A：`entities` 通用表 + type 字段（所有角色/地点/物品放一起）
- 优点：查询统一，关系管理简单
- 缺点：不同实体属性差异大，properties JSON 字段不类型安全

方案B：每个实体类型独立表（characters, locations, items...）
- 优点：类型安全，字段明确
- 缺点：表多，关系管理复杂

**你的倾向？**

### Q2: 章节内容存储

方案A：放数据库 TEXT 字段
- 优点：事务一致，查询方便
- 缺点：大文本拖慢查询

方案B：文件系统 + 数据库记录
- 优点：性能更好，版本控制方便
- 缺点：需要额外文件管理

方案C：数据库 + 向量化索引（混合）
- 优点：兼顾两者
- 缺点：最复杂

**你的倾向？**

### Q3: 向量数据库选型

方案A：sqlite-vec（开发用，零依赖）
- 优点：SQLite 扩展，不需要额外服务
- 缺点：性能不如专业向量库

方案B：pgvector（PostgreSQL 扩展）
- 优点：性能最好，生产级
- 缺点：需要 PostgreSQL

方案C：本地 Chroma/Qdrant
- 优点：专业向量功能丰富
- 缺点：额外进程

**你的倾向？**
