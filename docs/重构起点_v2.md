# NarrativeOS 重构起点 v2

> **重构铁律：LLM 是地基 → Agent 是墙 → 数据库是房间。先打地基，再砌墙，最后安门。**
> 
> 所有 AI 碰数据只有一条路：**提案 → 中转站 → 人点头 → 落库。**

---

## 一、重构宣言

### 1.1 旧方案的问题
- 39 张表、9 卷 25 章规格书，设计跑在代码前面
- 表先建好，Agent 后补，数据库成了无主空房
- 每个引擎各自设计 AI 写数据逻辑，维护灾难

### 1.2 新方案的核心转变
| 维度 | 旧方案 | 新方案 |
|------|--------|--------|
| 起点 | 数据库 schema 先设计 | LLM Client + Agent 基类先实现 |
| 数据生长 | 预设 39 张表 | Agent 需要存什么，表才长什么 |
| AI 写数据 | 各引擎各自实现 | 统一走 `ai_proposals` 中转站 |
| 协调 | 引擎间自主通信 | Orchestrator 薄层统一调度 |
| 验证方式 | 文档评审 | Phase 结束必须端到端可 demo |

## 二、三层架构总览

```
┌─────────────────────────────────────────┐
│  人（作者）— 唯一决策者                  │
│  拍板 / 驳回 / 要求修改 / 切换方向       │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│  Orchestrator（薄协调层）                │
│  - 接收人指令 → 路由到 Agent             │
│  - 接收 Agent 提案 → 写入中转站          │
│  - 等人审批 → 执行落库                   │
│  - 管理 session 上下文 + 并发锁          │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│  Agent 层（具体领域）                    │
│  ┌────────┐ ┌────────┐ ┌────────┐       │
│  │Hatching│ │ Studio │ │ World  │ ...   │
│  │ Agent  │ │ Agent  │ │ Agent  │       │
│  └────────┘ └────────┘ └────────┘       │
│       ↑ 继承 BaseAgent                   │
└─────────────────────────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│  LLM Client（统一封装）                  │
│  - 官方 SDK（Anthropic / OpenAI）        │
│  - 自动写 llm_logs                       │
│  - Token 计算 / 成本统计                 │
│  - 流式输出 / 降级重试                   │
└─────────────────────────────────────────┘
```

**铁律：Orchestrator 无决策权，Agent 无落库权，LLM Client 无业务逻辑。**

## 三、核心设计详解

### 3.1 LLM Client（第一层地基）

职责：统一封装官方 SDK，所有 LLM 调用必须经过这里。

```typescript
// 能力清单
interface LLMClient {
  // 流式调用（SSE）
  stream(model: ModelRef, messages: Message[], options: CallOptions): AsyncIterable<Chunk>
  
  // 非流式调用
  call(model: ModelRef, messages: Message[], options: CallOptions): Promise<Response>
  
  // Token 预算计算（tiktoken）
  estimateBudget(messages: Message[]): TokenBudget
  
  // 自动降级：主模型失败 → 备用模型
  callWithFallback(primary: ModelRef, fallback: ModelRef, messages: Message[]): Promise<Response>
  
  // 内部自动执行：写 llm_logs + 成本计算
}
```

**写入 `llm_logs` 的字段：**
- `model`, `prompt_tokens`, `completion_tokens`, `total_tokens`, `cost_usd`
- `latency_ms`, `status`, `error_message`
- `prompt_snapshot` / `response_snapshot`（>8KB 存 MinIO，`storage_ref`）
- `caller`（谁调的：HatchingAgent / Orchestrator / …）

---

### 3.2 Agent 基类（第二层墙基）

所有具体 Agent 继承此基类。基类实现 8 根桩，子类只写 2 个抽象方法。

```typescript
abstract class BaseAgent {
  // ========== 子类必须实现的（2个） ==========
  abstract name: string
  abstract getSystemPrompt(ctx: AgentContext): string      // 角色设定 + 记忆注入
  abstract parseProposal(response: string): Proposal         // 把 LLM 输出解析为标准提案

  // ========== 基类统一实现的（8根桩） ==========
  
  // 1. 执行入口：并发锁 + 链路追踪
  async execute(input: string, session: Session): Promise<Proposal>
  
  // 2. 工具调用（Tool Use）
  protected async callTool(toolName: string, params: unknown): Promise<unknown>
  
  // 3. Token 预算与自动裁剪
  private buildMessages(input: string, ctx: AgentContext): Message[]
  private cropMessages(messages: Message[], budget: TokenBudget): void
  
  // 4. 记忆注入（自动把 setting_items、项目配置、前序提案塞进 prompt）
  private async buildContext(session: Session): Promise<AgentContext>
  
  // 5. 提案 Schema 统一（所有子类产出必须符合 Proposal 结构）
  protected validateProposal(proposal: Proposal): ValidationResult
  
  // 6. 可观测性（执行链路追踪）
  protected tracer: Tracer
  
  // 7. 流式输出 + SSE 适配
  protected async streamLLM(messages: Message[]): Promise<string>
  
  // 8. 错误降级与重试
  protected async callWithRetry(messages: Message[]): Promise<string>
}
```

**Proposal 统一结构（所有 Agent 产出必须符合）：**

```typescript
interface Proposal {
  id?: string                    // 首次生成时为空，入库时生成
  type: string                   // 'world_setting' / 'character' / 'chapter_draft' / 'risk_alert' …
  title: string                  // 人可读标题
  content: unknown               // 提案具体内容（Agent 自定义结构）
  
  // 统一落库路由（Orchestrator 用）
  targetTable: string            // 'setting_items' / 'chapters' / 'world_events' / null（纯讨论）
  targetAction: 'insert' | 'update' | 'delete' | null
  targetId?: string              // update/delete 时指定
  payload?: unknown             // 要写入目标表的数据（JSONB-ready）
  
  // 审批模式
  approvalMode: 'manual' | 'auto' | 'threshold'  
  impactScore?: number           // 影响范围分，threshold 模式用
  
  // 元数据
  sourceAgent: string            // 哪个 Agent 产的
  sessionId: string
  parentId?: string              // 修改版本指向原提案
  version: number               // 默认 1
}
```

### 3.3 Orchestrator（薄协调层）

**职责只有四个：路由 → 调用 → 中转 → 执行。**

```typescript
class Orchestrator {
  // 当前阶段只有 HatchingAgent，后期加路由表
  private agents: Map<string, BaseAgent>
  
  async handle(input: UserInput, session: Session): Promise<OrchestratorResult> {
    // 1. 路由：判断调哪个 Agent（现在只有 HatchingAgent）
    const agent = this.resolveAgent(input.intent)
    
    // 2. 调用 Agent 生成提案
    const proposal = await agent.execute(input.content, session)
    
    // 3. 写入中转站（所有 AI 产出必须先落这里）
    const saved = await this.saveProposal(proposal)
    
    // 4. 如果是 auto 模式且低风险，直接执行；否则等人审批
    if (proposal.approvalMode === 'auto' && (proposal.impactScore || 0) < 10) {
      await this.executeProposal(saved)
      return { type: 'auto_executed', proposal: saved }
    }
    
    // 默认：推给人，等审批
    return { type: 'pending_approval', proposal: saved }
  }
  
  // 人审批后调用
  async approve(proposalId: string, verdict: 'approve' | 'reject' | 'revise'): Promise<void> {
    if (verdict === 'approve') {
      await this.executeProposal(await this.getProposal(proposalId))
    } else if (verdict === 'revise') {
      // 生成新版本，parentId 指向原提案
      await this.createRevision(proposalId)
    }
    // reject：只改状态，不执行
  }
  
  // 统一执行落库（所有表只有这一个入口被 AI 写入）
  private async executeProposal(proposal: Proposal): Promise<void> {
    if (!proposal.targetTable) return  // 纯讨论，不落库
    
    // 按 targetTable + targetAction + payload 执行
    await db[proposal.targetTable][proposal.targetAction](proposal.payload)
    
    // 触发后续：如果是 setting_items，更新 Agent 上下文缓存
    if (proposal.targetTable === 'setting_items') {
      await this.invalidateContextCache(proposal.sessionId)
    }
  }
}
```

**关键纪律：**
- Orchestrator 不决策（选哪个方案、怎么改，都是人的事）
- Orchestrator 不生成内容（只调度 Agent，不碰 LLM）
- Orchestrator 是所有 AI 数据写入的**唯一执行入口**

## 四、数据库起点（6 + 1 张表）

先 6 张表跑通孵化流程。第 7 张（`chapters`）等 Phase 3 再加。

```sql
-- 1. projects：项目主表
CREATE TABLE projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL,
  genre TEXT NOT NULL,
  style TEXT,
  target_words INTEGER,
  status TEXT NOT NULL DEFAULT 'hatching',  -- hatching / active / paused / completed
  genre_contract JSONB,
  world_bible JSONB,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 2. sessions：会话表
CREATE TABLE sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects(id),
  type TEXT NOT NULL,  -- hatching / chapter_gen / world_discuss …
  title TEXT,
  status TEXT NOT NULL DEFAULT 'active',  -- active / closed
  context_snapshot JSONB,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  closed_at TIMESTAMPTZ
);

-- 3. discussions：消息记录
CREATE TABLE discussions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects(id),
  session_id UUID NOT NULL REFERENCES sessions(id),
  role TEXT NOT NULL CHECK (role IN ('user', 'assistant')),
  content TEXT NOT NULL,
  proposal_id UUID,
  metadata JSONB,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 4. ai_proposals：AI 产出中转站（核心）
CREATE TABLE ai_proposals (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects(id),
  session_id UUID NOT NULL REFERENCES sessions(id),
  
  -- 提案内容
  type TEXT NOT NULL,
  title TEXT NOT NULL,
  content JSONB NOT NULL,
  
  -- 统一落库路由（新增）
  target_table TEXT,           -- 'setting_items' / 'chapters' / 'world_events' …
  target_action TEXT CHECK (target_action IN ('insert', 'update', 'delete')),
  target_id UUID,              -- update/delete 时指定
  payload JSONB,               -- 要写入目标表的数据
  
  -- 审批模式（新增）
  approval_mode TEXT NOT NULL DEFAULT 'manual' CHECK (approval_mode IN ('manual', 'auto', 'threshold')),
  impact_score INTEGER,        -- 影响范围分 0-100
  
  -- 状态机
  status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'approved', 'rejected', 'superseded')),
  version INTEGER NOT NULL DEFAULT 1,
  parent_id UUID REFERENCES ai_proposals(id),
  rejection_note TEXT,
  approved_at TIMESTAMPTZ,
  
  -- 来源追踪
  source_agent TEXT NOT NULL,   -- 哪个 Agent 产的：'hatching' / 'studio' …
  
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 5. setting_items：设定集
CREATE TABLE setting_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects(id),
  proposal_id UUID REFERENCES ai_proposals(id),
  type TEXT NOT NULL,  -- character / location / faction / rule / worldview / item …
  name TEXT NOT NULL,
  summary TEXT NOT NULL,  -- 用于 prompt 快速注入
  content JSONB NOT NULL,
  status TEXT NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'confirmed', 'archived')),
  tags TEXT[],
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 6. llm_logs：LLM 调用日志
CREATE TABLE llm_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID REFERENCES projects(id),
  session_id UUID REFERENCES sessions(id),
  caller TEXT NOT NULL,         -- 'hatching_agent' / 'orchestrator' / …
  caller_ref_id UUID,             -- 关联 proposal_id 或 discussion_id
  caller_ref_type TEXT,
  model TEXT NOT NULL,
  prompt_tokens INTEGER,
  completion_tokens INTEGER,
  total_tokens INTEGER,
  cost_usd NUMERIC(10, 6),
  latency_ms INTEGER,
  status TEXT NOT NULL CHECK (status IN ('success', 'error', 'timeout')),
  error_message TEXT,
  prompt_snapshot JSONB,
  response_snapshot JSONB,
  storage_ref TEXT,               -- >8KB 时存 MinIO，这里记路径
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

**新增索引（最小集）：**
```sql
CREATE INDEX idx_ai_proposals_session ON ai_proposals(session_id, status, created_at DESC);
CREATE INDEX idx_ai_proposals_source ON ai_proposals(source_agent, status);
CREATE INDEX idx_setting_items_project_type ON setting_items(project_id, type);
CREATE INDEX idx_discussions_session ON discussions(session_id, created_at DESC);
CREATE INDEX idx_llm_logs_project ON llm_logs(project_id, created_at DESC);
```

## 五、Phase 0：地基（第 1-2 天）

**目标：六张表建好，LLM Client 能调通，Agent 基类骨架能继承。**

### 5.1 任务清单

```
□ 初始化 PostgreSQL 数据库，执行上面 6 张表 DDL
□ 实现 LLM Client 封装层
  □ 集成 Anthropic SDK（主）+ OpenAI SDK（备）
  □ 流式输出封装（AsyncIterable）
  □ Token 计算（tiktoken 或 SDK 自带）
  □ 自动写 llm_logs（每次调用完同步写）
  □ 降级逻辑：Anthropic 失败 → 换 OpenAI
□ 实现 BaseAgent 基类（8 根桩）
  □ execute() 入口：并发锁（Redis 或内存 Map）+ 伪链路追踪（console.log 版）
  □ buildContext()：读取 session + project + 已确认 setting_items
  □ buildMessages()：system prompt + history + user input
  □ cropMessages()：超预算时截断历史（保留 system + 最近 N 轮）
  □ streamLLM()：调用 LLM Client
  □ validateProposal()：校验 Proposal 结构完整
  □ 错误降级：超时/拒单时抛异常， Orchestrator 捕获后重试或换人
□ 实现第一个具体 Agent：HatchingAgent
  □ 继承 BaseAgent
  □ 实现 getSystemPrompt()：孵化专员角色 + 项目信息注入
  □ 实现 parseProposal()：解析 LLM JSON 输出为 Proposal 结构
  □ 支持产出 type='world_setting' / 'character' / 'rule' / 'style' 的提案
□ Orchestrator 最小实现
  □ 硬编码路由：所有 intent → HatchingAgent
  □ handle()：调用 Agent → saveProposal → 返回 pending_approval
  □ approve()：approve 时按 target_table 写入 setting_items
□ API 接口（Hono / Fastify）
  □ POST /projects — 创建项目（自动建 hatching session）
  □ POST /sessions/:id/messages — 人发消息 → Orchestrator.handle()
  □ POST /proposals/:id/verdict — 人审批（approve/reject/revise）
  □ GET /sessions/:id/discussions — 拉取对话历史
  □ GET /projects/:id/setting-items — 拉取已确认设定
```

### 5.2 验收标准

```
1. curl 能创建项目
2. curl 能发消息 → 看到 AI 在 thinking（流式输出）
3. AI 返回的内容解析为 Proposal 入库 ai_proposals
4. 人在接口点 approve → 数据自动写入 setting_items
5. 查看 llm_logs 有完整调用记录
6. 查看 discussions 有完整的 user/assistant 对话历史
```

## 六、Phase 1：孵化流程（第 3-5 天）

**目标：人输入书名+类型+风格+字数，AI 介入讨论设定，人能反复沟通、修改、拍板。**

### 6.1 新增能力

```
□ HatchingAgent 支持多轮对话上下文（session discussions 历史注入）
□ Agent 产出提案时自动分析需要哪些设定项（世界观/力量体系/主角/势力…）
□ 支持 request_revision：人说"再改一版"→ 生成新版本（parent_id 关联）
□ 设定集查看接口（按 type 分组，显示状态 draft/confirmed）
□ project status 流转：hatching → active（手动触发，表示设定集锁定）
```

### 6.2 验收标准

```
1. 创建一个修仙项目，AI 自动推送第一批世界核心设定提案
2. 人 approve 3 个提案 → setting_items 有 3 条 confirmed 数据
3. 人对第 4 个提案说 revise → ai_proposals 多出 version=2 的新记录
4. 人连续聊 20 轮，上下文不丢失，AI 记得之前确认过什么
5. 人点"设定集锁定"→ project.status 变为 active
```

## 七、Phase 2：设定集深化（第 6-8 天）

**目标：锁定后仍能深化设定，但不影响已确认项；支持标签检索和快速注入。**

```
□ setting_items 支持在锁定后新增（新提案 → 新 type/新 name）
□ 已 confirmed 的 item 修改需走 revise 流程（原 item 标记 archived，新建 draft）
□ HatchingAgent 读取 confirmed setting_items 摘要注入 prompt
□ 支持按 tags 检索设定（快速定位）
□ 世界圣经 world_bible JSONB 生成（由 AI 基于所有 confirmed items 总结）
```

## 八、Phase 3：第一章生成（第 9-12 天）

**目标：设定集锁定后，生成第一章正文草稿。**

```
□ 新增 chapters 表（第 7 张表）
□ 新增 StudioAgent（继承 BaseAgent）
□ StudioAgent system prompt 注入：world_bible + confirmed setting_items 摘要 + 章节方向
```

## 十一、开发纪律（不可违反）

1. **Agent 无落库权**：所有 Agent 产出必须经 Orchestrator → ai_proposals → 人审批 → 才落库
2. **LLM 调用全记录**：任何绕过 llm_logs 的调用都是 bug
3. **Proposal 结构统一**：任何 Agent 产出不符合 Proposal 接口都是 bug
4. **Phase 验收制**：每 Phase 必须有 curl 命令能走完端到端，否则不进入下一 Phase
5. **表按需生长**：Phase N 之前不要建 Phase N+1 的表
6. **前端滞后原则**：后端没跑通前不写前端
7. **人排版 AI 执行**：任何 AI 自主推进流程的代码都是违反第一公理

## 十二、一句话总结

> **六张表，一个 Agent，一层 Orchestrator，一套 LLM Client。**
> 
> **跑通"人创建项目 → AI 推送设定 → 人拍板确认 → 设定入库"这条线，再谈世界引擎。**
