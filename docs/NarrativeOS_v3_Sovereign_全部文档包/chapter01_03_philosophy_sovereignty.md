# NarrativeOS v3.0 Sovereign 设计文档

---

# 第一章：项目概述与第一公理

## 1.1 项目定位

### 1.1.1 目标用户画像

NarrativeOS 服务于**单机作者（Solo Author）**——即独立承担 100 万字以上长篇网文创作任务的写作者。这类作者具有如下特征：

- **创作规模**：正在或将要进行单部作品 100 万至 500 万字的超长篇叙事创作，世界观设定跨度大、人物关系网络复杂、剧情线索多线交织。
- **工作模式**：独自一人承担全部创作职能——世界观架构师、角色设计师、情节编排者、正文撰写者、质量把控者、节奏调度师。
- **核心痛点**：随着篇幅增长，维护内部一致性（人设不崩、战力不崩、时间线不矛盾、伏笔不丢失）的认知负担呈指数级上升；创作中频繁需要"全局搜索记忆"以确认某角色三百年前的台词是否与当前行为冲突，或某功法在三百章前的设定是否允许当前剧情展开。
- **技术接受度**：对 AI 辅助创作持开放态度，但**拒绝任何形式的自动发表或自动决策**——每一行文字的最终裁决权必须归属作者本人。
- **审美主权**：拥有强烈的个人风格与叙事追求，系统生成的任何内容若不符合其审美标准，必须能够被直接否决并要求重新生成，且不因此受到系统层面的隐性惩罚（如生成质量下降、选项变少等）。

**明确排除的用户群体**：

- 寻求"一键成书"全自动写作工具的非创作者；
- 以短篇小说（<10 万字）为主要创作形态的写作者；
- 协作创作团队（多人共同创作同部作品），NarrativeOS 的权限模型基于单一人类主权者设计；
- 需要商业化出版工作流（排版、ISBN 申请、多渠道分发）的传统出版作者。

### 1.1.2 系统形态定义：导演工作室隐喻

NarrativeOS 不将自己定位为"写作软件"，也不定位为"AI 写作工具"。它的形态是一个**围绕作者运转的虚拟创作团队**——"导演工作室"。

在这个工作室中，作者是唯一的**导演**。导演不亲自操作摄影机、不亲自搭建布景、不亲自撰写分镜脚本，但每一个关键决策点——"这个镜头怎么拍""这场戏的情绪基调是什么""这个角色此刻的眼神应该怎么呈现"——都必须由导演拍板。工作室中的其他成员负责将导演的创作意图转化为可执行方案，并在执行前再次提交导演确认。

这种形态意味着：

- **作者与系统的关系**不是"操作者与工具"，而是**"导演与团队"**——系统是有能动性（agency）的协作者，但这个能动性受到严格的权力边界约束。
- **系统必须具备理解能力**——它需要理解作者的叙事意图、审美偏好、风格特征，并在后续提案中保持这种理解的一致性。
- **系统必须具备记忆能力**——它需要记住整部作品的全部细节（世界观、角色、事件、伏笔），并在任何时候能够回答作者的询问。
- **系统必须具备推演能力**——给定一个创作决策，系统需要能够推演该决策对世界状态的影响，并量化这些影响（代价、收益、风险）。

但系统**绝不具备裁决能力**。理解、记忆、推演，一切服务于为导演提供更充分的信息和更高质量的提案，最终决策权始终锁定在人类手中。

### 1.1.3 核心价值主张

#### 为什么需要 NarrativeOS？

长篇网文创作的**一致性维护问题**是一个经典的认知复杂度爆炸问题。一部 300 万字的作品可能包含：

- 数百个有名有姓的角色，每个角色有自己的性格弧光、能力参数、社会关系、台词风格；
- 数千个设定条目，涵盖世界规则、功法体系、势力分布、地理格局、历史事件；
- 数万条情节线索，主线、支线、暗线交织，伏笔与回收关系构成有向无环图；
- 数十万处细节引用——某角色在第三章说过什么，在第五百章的行动是否与之矛盾。

**纯文本编辑器**（如 Word、Scrivener）提供了良好的书写体验，但完全不提供一致性维护能力。作者必须依赖自己的大脑记忆或外部笔记来追踪全部信息，这在超长篇幅下是不可持续的。

**现有 AI 写作助手**（如各类 GPT 写作插件）提供了生成能力，但存在三个根本缺陷：

1. **叙事裁决权越界**：自动生成正文并直接呈现，将"生成"与"确认"两个步骤混淆，潜移默化地剥夺了作者对叙事走向的掌控感。
2. **状态记忆缺失**：多数 AI 写作工具是无状态的或仅有短期上下文，无法维护跨越数百万字的全局世界观和角色状态。
3. **风格一致性脆弱**：生成的内容质量波动大，对作者个人风格的模仿深度不足，且不同会话间的风格一致性无法保证。

NarrativeOS 的核心价值在于：**在维护作者绝对主权的前提下，将超长篇创作中机器擅长的工作（全局记忆、逻辑推演、一致性检查、方案生成）从人类大脑中卸载到系统中，让作者将认知资源集中在只有人类能做的创造性工作上**。

具体而言，NarrativeOS 承诺提供三类价值：

**V1. 全局记忆外化**：将整部作品的全部结构化信息（世界观、角色、事件、设定）持久化存储在系统中，任何时候作者可以询问"某角色在第三章说过什么""某功法在两百章前的设定是什么"，系统在亚秒级返回精确答案。作者不再需要依赖大脑记忆或外部笔记。

**V2. 一致性自动守护**：系统在每次创作活动后自动执行一致性检查——新写的内容是否与现有设定矛盾？是否与人设冲突？是否破坏了已建立的战力体系？风险被标记并呈现给作者，附带修复建议，但**修复决策由作者做出**。

**V3. 创作提案加速**：当作者需要推进剧情时，系统基于对当前世界状态、角色心理、叙事节奏的完整理解，生成多个高质量的创作提案（情节走向、对话方案、场景描写），每个提案附带代价分析和风险评估，由作者选择或否决。

### 1.1.4 外部 KPI 与内部指标

#### 外部 KPI（唯一）

**平台的读者看得爽，一直追更，催更。**

这是系统存在的唯一外部验证标准。NarrativeOS 不追求生成速度、不追求自动完成率、不追求 AI 输出字数。它唯一关心的是：通过赋能作者，让最终呈现给读者的作品质量更高、更新更稳定、故事更吸引人。

这个 KPI 无法被系统直接度量（系统不直接观测读者行为），但它是所有设计决策的最终指向。任何功能若不能直接或间接地服务于"让读者看得爽"这个目标，就不应该被纳入系统。

#### 内部指标（辅助仪表）

系统运行过程中产生一系列内部度量，用于衡量各子系统的运行质量，但这些指标**绝不替代作者的确认**：

| 内部指标 | 定义 | 用途 | 约束 |
|----------|------|------|------|
| **语言质感评分** | 生成正文在作者风格一致性、文笔流畅度、画面感、节奏感上的综合评分（0-100） | 帮助作者快速评估不同提案的质量差异 | 评分仅作排序参考，作者有权选择低分方案或重写全部 |
| **战力合法性评分** | 新剧情对世界战力体系的冲击程度量化（0-100，100 为完全合法） | 标记战力崩坏风险 | 系统仅标记风险，不阻止作者创作"越级"剧情 |
| **追更欲模拟指数** | 基于叙事节奏模型，预测当前章节断点对读者"追更欲望"的影响（0-100） | 辅助作者优化断章节奏 | 模拟结果仅供参考，作者可按自己的节奏叙事 |
| **混沌熵值** | 当前世界状态相对初始设定的偏离程度（详见 1.3.1） | 帮助作者理解叙事演进带来的系统性变化 | 高熵值不是错误，是创作的自然结果 |
| **心流熵值** | 作者在创作过程中的心流状态度量（详见 1.3.3） | 辅助 Flow Guardian 做出模式切换建议 | 系统仅提醒，不强制切换 |

**关键原则**：所有内部指标都服务于"为作者提供信息"，而非"替代作者决策"。任何将内部指标硬编码为自动决策依据的行为都是对第一公理的违反。

### 1.1.5 系统边界

#### NarrativeOS 做什么

- 维护长篇叙事作品的结构化知识图谱（世界观、角色、事件、设定、伏笔）
- 在作者创作过程中提供实时的知识查询和上下文检索
- 基于当前世界状态生成创作提案（情节走向、对话、场景）
- 对作者创作的内容执行一致性检查并报告风险
- 追踪和维护叙事势能（详见 1.3.2）
- 管理创作工作流（章节规划、写作、审核、固化）
- 监测作者心流状态并提供模式切换建议
- 记录完整的创作审计日志

#### NarrativeOS 不做什么

- **不自动生成并确认正文**：所有生成内容都是提案，必须经过作者确认才能进入作品
- **不修改已固化的章节**：已固化的章节只有通过作者的明确解冻指令才能修改
- **不替代作者进行创意决策**：故事走向、角色命运、情节节奏由作者决定
- **不提供读者端功能**：没有评论系统、投票系统、读者社区
- **不提供发布功能**：不直接对接任何网文平台 API
- **不提供文本格式排版**：输出纯文本，排版由外部工具处理
- **不做"最佳方案"评判**：系统可以提供排序，但"哪个方案最好"的最终判断权在作者

### 1.1.6 差异化定位矩阵

| 维度 | 纯文本编辑器 | AI 写作助手 | NarrativeOS |
|------|------------|-----------|-------------|
| 全局知识管理 | 无 | 弱（上下文窗口限制） | **强（持久化知识图谱）** |
| 一致性检查 | 无 | 无或弱 | **强（多维度自动检查）** |
| 生成能力 | 无 | **强（快速生成）** | 强（提案式生成） |
| 叙事主权 | 完全归属作者 | **存在越界风险** | **绝对锁定人类** |
| 风格一致性 | N/A | 弱（会话间不稳定） | **强（持续学习作者风格）** |
| 长文本支持 | 好 | 差（上下文限制） | **好（架构为百万字级设计）** |
| 心流保护 | 无 | 无 | **有（三模式控制台）** |

## 1.2 第一公理

### 1.2.1 公理的哲学基础

**公理一：作者不在引擎内部，作者在引擎之上。**

这条公理不是技术决策，而是伦理决策。它源于一个根本判断：**叙事创作是人类最深层的精神活动之一，叙事主权是不可让渡的人类权利。**

在任何由人类与机器组成的协作系统中，存在一个权力分配的光谱：

- **左端**：人类完全控制，机器是被动工具（如打字机）
- **右端**：机器自主运行，人类是被动监督者（如全自动内容农场）
- **中间地带**：人机协作，权力动态分配

NarrativeOS 选择的光谱位置是：**极度偏左，但带有高度智能的辅助**。系统不是被动工具——它理解、记忆、推演、提案——但它绝不做出任何不可撤销的叙事决策。所有不可撤销操作（固化章节、修改世界状态、确认正文进入作品）必须由人类明确触发。

这条公理的伦理基础包括三个层次：

**创作者主权（Creator Sovereignty）**：创作者对其作品拥有不可剥夺的完整权利。在 NarrativeOS 的语境下，这意味着创作者对作品的每一个字节都拥有否决权。系统可以建议、可以提醒、可以标记风险，但**不能越过创作者的意志将任何内容写死**。

**认知自主性（Cognitive Autonomy）**：创作者在创作过程中的每一个决策——即使是一个"糟糕"的决策——都是其创作表达的一部分。系统不应该以"优化"为名，通过设计上的便利性偏差（如将某个选项放在更显眼的位置、或让某个选项更容易点击）来引导创作者做出"系统认为更好"的决策。

**问责明确性（Accountability Clarity）**：当一部作品呈现给读者时，必须能够明确回答"这是谁的决定"。NarrativeOS 确保每一个进入作品的决策都有清晰的人类问责锚点——在审计日志中，每一个状态变更都可以追溯到一个人类事件。

### 1.2.2 "作者在引擎之上"的技术含义

"作者在引擎之上"不是比喻，而是一种严格的架构约束。在技术层面，这意味着：

**1. 架构层级约束**

在 NarrativeOS 的分层架构中，作者交互界面位于最高层，协调层（MOU 状态机）位于第二层。协调层的设计遵循一个铁律：**每一个 `waiting_*` 状态都是阻塞状态，必须等待明确的人类事件才能转移到下一个状态。**

协调层不允许：
- 在没有人类事件的情况下自动从 `waiting_author_decision` 转移到任何执行状态
- 设置超时自动推进机制（如"30 秒无响应自动选择第一个选项"）
- 在没有人类确认的情况下将任何内容标记为"已确认"或"已固化"

**2. 数据流向约束**

所有涉及作品内容的不可逆变更（章节固化、世界状态修改、设定变更）的数据流都遵循"人类事件触发"原则：

```
人类事件 → 协调层状态转移 → 执行层操作 → 数据库持久化
```

没有人类事件，就没有状态转移；没有状态转移，就没有执行；没有执行，就没有数据变更。

**3. 决策权映射**

| 决策类型 | 决策权归属 | 系统角色 |
|----------|-----------|----------|
| 确认正文进入作品 | **人类专属** | 仅提供提案 |
| 选择情节走向 | **人类专属** | 仅提供选项与影响分析 |
| 修改世界观设定 | **人类专属** | 仅标记修改的影响范围 |
| 冻结/解冻章节 | **人类专属** | 仅执行指令并记录 |
| 知识查询 | 系统自动 | 无需人类确认 |
| 一致性检查 | 系统自动 | 检查结果标记风险，人类决定是否修复 |
| 提案生成 | 系统自动 | 生成方案，人类选择 |
| 心流监测 | 系统自动 | 监测并建议模式切换，人类决定是否切换 |

### 1.2.3 叙事裁决权的明确定义

**叙事裁决权（Narrative Adjudication Right）**是作者在 NarrativeOS 中拥有的最高权力，指对以下内容做出最终决定并使之生效的专属权利：

**属于叙事裁决权的事项（必须人类执行）：**

1. **正文确认权**：决定哪些文字构成作品的正式内容。系统生成的提案在获得作者确认前，只是临时内容，不进入作品。
2. **情节走向决定权**：选择故事如何发展。系统可以提供多个情节分支的提案及影响分析，但选择哪个分支由作者决定。
3. **世界状态变更权**：修改世界观设定、角色属性、势力格局等持久化状态。系统可以标记变更的影响，但变更指令必须由作者发出。
4. **规则例外批准权**：当创作需要突破已建立的世界规则时（如"虽然设定上说这个功法需要十年修炼，但我现在想让主角在三章内突破"），批准权归作者。
5. **固化/解冻权**：决定章节何时从"可编辑"状态进入"已固化"状态，或从"已固化"状态解冻重新编辑。
6. **风险接受权**：当谏官标记风险后，作者可以选择接受风险（"我知道这里有逻辑漏洞，但读者不会注意到，我选择保留"）或要求修复。
7. **风格主权**：系统必须学习并模仿作者的风格，但作者有权随时否定系统的风格模仿结果，并要求重新调整。

**不属于叙事裁决权的事项（系统可自主执行）：**

1. **知识检索**：响应作者的知识查询（如"张三在第三章说过什么"）
2. **一致性扫描**：对作者确认的内容自动执行一致性检查
3. **影响推演**：给定假设条件，推演对世界状态的影响
4. **提案生成**：基于当前上下文生成创作提案
5. **指标计算**：计算各项内部指标（语言质感、战力合法性、混沌熵值、心流熵值）
6. **审计日志记录**：自动记录系统运行中的所有事件

### 1.2.4 人类确认点的完整清单

以下列出了 NarrativeOS 中**所有必须经过人类确认才能继续**的关键节点。协调层的状态机中，每个节点对应一个 `waiting_*` 状态：

| # | 确认点名称 | 对应状态 | 说明 |
|---|-----------|----------|------|
| 1 | **提案选择确认** | `waiting_author_selection` | 系统生成多个创作提案后，作者必须选择其中一个、要求重新生成、或选择"听我的"（自行创作） |
| 2 | **正文确认** | `waiting_author_approval` | 作者对系统生成或自行撰写的正文进行确认，确认后进入质检流程 |
| 3 | **质检结果确认** | `waiting_qa_review` | 质检系统标记问题后，作者确认修复方案或直接批准通过 |
| 4 | **风险标记确认** | `waiting_risk_resolution` | 谏官标记风险并提供修复建议后，作者选择接受风险、按建议修复、或自行修复 |
| 5 | **章节固化确认** | `waiting_freeze_confirmation` | 作者确认将当前章节从"可编辑"状态固化为"已固化"状态 |
| 6 | **世界状态变更确认** | `waiting_state_change_confirmation` | 涉及世界状态（设定、角色属性、势力格局）的变更在生效前需作者确认 |
| 7 | **混沌种子确认** | `waiting_chaos_seed_approval` | 当混沌种子承诺机制触发时，作者确认混沌参数后系统才执行注入 |
| 8 | **模式切换确认** | `waiting_mode_switch_confirmation` | Flow Guardian 建议切换心流模式时，作者确认后才执行切换 |
| 9 | **解冻确认** | `waiting_thaw_confirmation` | 当作者请求修改已固化章节时，系统展示解冻影响分析，作者确认后才执行解冻 |
| 10 | **批量修改确认** | `waiting_batch_modification_confirmation` | 当某一设定变更影响多部章节时，系统展示影响范围，作者确认后才执行批量修改 |
| 11 | **叙事势能释放确认** | `waiting_potential_release_approval` | 当系统检测到高势能蓄积点时，作者确认后才辅助释放 |
| 12 | **情报消费确认** | `waiting_intel_consumption_confirmation` | 工作室引擎向世界引擎请求关键情报时，作者确认消耗参数后才执行查询 |

### 1.2.5 系统自主行为的明确边界

以下行为由系统自动执行，**不需要**人类确认，但全部记录在审计日志中：

**自主执行（无需确认）：**

| 行为 | 触发条件 | 输出 |
|------|----------|------|
| 知识图谱查询 | 作者发起查询请求 | 查询结果 |
| 一致性自动扫描 | 每次正文确认后 | 风险标记列表（待人类确认处理） |
| 指标计算 | 定时或事件触发 | 各项内部指标更新 |
| 提案生成 | 作者请求或流程触发 | 提案列表（待人类选择） |
| 审计日志写入 | 所有系统事件 | 不可篡改的审计记录 |
| 世界状态快照 | 每次固化前自动创建 | 快照记录（用于回滚） |
| 心流状态监测 | 持续后台运行 | 心流熵值更新 |
| 提案预加载 | 系统空闲时 | 预热可能的提案（减少等待时间） |

**约束**：所有自主行为的输出都是**信息性**的（提供数据、标记风险、生成选项），而不是**决策性**的（修改状态、确认内容、推进流程）。流程推进的每一步都必须有一个人类事件作为触发器。

### 1.2.6 违反公理的检测机制

NarrativeOS 内置公理违反检测系统，用于检测和防止系统越权：

**检测维度**：

1. **状态机异常检测**：监控协调层状态机的状态转移日志，检测是否存在没有对应人类事件的 `waiting_* → executing` 转移。
2. **数据变更审计**：监控数据库变更日志，检测是否存在没有人类确认锚点的作品正文变更、世界状态变更、固化状态变更。
3. **自动决策扫描**：定期检查系统配置和代码路径，扫描是否存在自动选择、自动确认、自动推进的逻辑分支。
4. **超时绕过检测**：检测是否存在任何形式的超时自动推进机制。

**检测响应**：

当检测到公理违反时：
1. 立即触发**系统冻结**（System Freeze）：所有引擎暂停执行，当前操作被中断
2. 向作者发送**越权警报**，包含违规详情、涉及的操作、建议的恢复方案
3. 将违规事件写入**不可篡改的违规日志**（独立于常规审计日志）
4. 进入**手动恢复模式**，需要作者明确确认后才能恢复系统运行

详见第三章 3.5 节"越权处理和恢复"的完整规格。

## 1.3 三组核心悖论

长篇叙事创作中存在三组深层张力。它们不是可以被"解决"的问题，而是需要被**管理**的辩证关系。NarrativeOS 为每组悖论设计了一套动态平衡机制。

### 1.3.1 混沌层：安全感与奇迹的张力

#### 悖论描述

**A 端：作者需要安全感。** 创作是一个高不确定性活动。作者需要相信系统不会丢失她的工作、不会擅自修改她的文字、不会在她不知情的情况下让故事走向她未曾批准的方向。这种安全感是创作的基础心理条件。

**B 端：叙事需要奇迹。** 伟大的叙事往往诞生于受控的意外——一个角色突然做出了连作者都未曾预料的举动，一个伏笔在意想不到的地方回收，一个世界规则在极限状态下展现出新的维度。如果系统过度追求确定性，它会扼杀叙事的自发性和惊喜感。

#### 技术方案：混沌种子承诺（Chaos Seed Commitment）

混沌种子承诺是一种"有约束的随机注入"机制。它允许系统在作者的授权下，在叙事中引入受控的混沌元素，但这些混沌元素的注入遵循严格的协议：

**核心原则**：

- **预先授权**：混沌的注入必须在执行前获得作者明确授权
- **参数透明**：混沌的参数（范围、强度、影响域）在注入前完整展示给作者
- **事后审计**：混沌注入后的全部影响可被追溯审计
- **不可回溯**：一旦混沌注入并产生级联影响，不能简单地"撤销"——因为叙事已经因此改变，撤销会破坏因果一致性

**算法流程**：

```
阶段 1：混沌机会识别
├── 输入：当前世界状态 W，叙事势能分布 P，作者近期决策模式 D
├── 世界引擎分析是否存在"混沌 fertile ground"（高势能、多不确定性、低约束区域）
├── 若存在，生成混沌种子提案 C = {type, range, intensity, domain, rationale}
└── 提案进入等待作者确认状态

阶段 2：作者授权
├── 系统将 C 的完整参数展示给作者
│   ├── type: 混沌类型（角色突变 / 事件分支 / 设定揭示 / 关系重构 / 规则突破）
│   ├── range: 影响范围（单场景 / 单章节 / 跨章节 / 世界观级）
│   ├── intensity: 混沌强度（1-10，1 为轻微扰动，10 为结构性冲击）
│   ├── domain: 影响域（具体涉及的角色、设定、势力）
│   └── rationale: 系统建议此混沌的叙事理由
├── 作者有三个选择：
│   ├── 批准（进入阶段 3）
│   ├── 修改参数后批准（调整任一参数后进入阶段 3）
│   └── 拒绝（混沌种子废弃，不影响任何状态）
└── 此阶段阻塞等待人类事件

阶段 3：混沌注入执行
├── 若作者批准，系统执行混沌注入
├── 混沌引擎根据 C 的参数生成具体的混沌事件 E
├── E 被写入世界状态，作为待确认提案呈现给作者
└── 级联影响分析启动

阶段 4：级联影响分析
├── 世界引擎推演 E 对世界状态的级联影响
│   ├── 直接影响：E 直接修改的状态变量
│   ├── 一级间接影响：因直接影响的传播效应
│   ├── 二级+ 影响：更深层的传播（可选，取决于性能约束）
│   └── 一致性风险评估：E 是否与现有设定矛盾
├── 影响分析报告 I 呈现给作者
└── 作者确认接受影响、要求调整、或放弃混沌（进入回滚流程）

阶段 5：固化与审计
├── 作者确认后，混沌及其级联影响被写入世界状态
├── 混沌记录被写入审计日志（不可删除）
├── 混沌参数、影响范围、作者授权凭证被永久保存
└── 事后可在任何时候查询"此处的叙事走向源于哪次混沌注入"
```

**混沌类型定义**：

| 类型 | 描述 | 示例 | 强度范围 |
|------|------|------|----------|
| **角色突变** | 角色展现出与既有性格不一致的行为 | 一个冷静的角色突然暴怒 | 3-7 |
| **事件分支** | 故事走向与最可能的线性发展偏离 | 主角错过了一个关键信息 | 4-8 |
| **设定揭示** | 已存在的设定展现出未曾预见的维度 | 一个看似普通的物品其实是神器 | 5-9 |
| **关系重构** | 两个角色间的关系性质发生改变 | 宿敌原来是失散兄弟 | 6-10 |
| **规则突破** | 世界规则在特定条件下被突破 | 某个"不可能"的事件发生了 | 7-10 |

**混沌承诺的安全约束**：

1. **累计混沌预算**：每个叙事篇章（如一部小说的一个"卷"）有一个作者设定的混沌预算上限。系统不会提出超出预算的混沌提案。
2. **混沌冷却期**：同一影响域在两次混沌注入之间必须有最小间隔（默认 5 章）。
3. **一致性硬约束**：混沌注入后，世界引擎必须确认混沌与核心设定（世界底层规则、已固化章节的关键事件）不矛盾。若存在矛盾，混沌必须在作者授权下修改核心设定后才能注入，或者混沌被降级为"幻觉/误解"（即角色以为发生了某事，但实际并非如此）。
4. **作者的终极否决权**：在混沌流程的任何阶段，作者都可以中止流程，废弃混沌种子，已进行的任何临时推演不被持久化。

#### 数学模型：混沌熵值

混沌熵值 $H_c$ 度量当前世界状态相对于初始设定的偏离程度：

$$H_c = \sum_{i} w_i \cdot \delta_i$$

其中：
- $i$ 遍历所有被混沌注入修改过的状态变量
- $\delta_i$ 是第 $i$ 个变量的偏离幅度（相对于初始设定的标准化偏离）
- $w_i$ 是该变量在叙事重要性上的权重（由世界引擎根据该变量与其他状态变量的关联度计算）

混沌熵值不是错误指标，而是**叙事演进度量**。随着故事推进，$H_c$ 自然增长——这是创作过程的自然结果。系统记录 $H_c$ 的历史曲线，用于帮助作者理解"我的故事现在偏离初始设想有多远"。

### 1.3.2 先例层：严谨与灵活的张力

#### 悖论描述

**A 端：规则需严谨。** 长篇叙事依赖于读者的信任——读者相信作者对世界的设定是内在一致的，前面的伏笔会在后面回收，建立的规则不会在关键时刻被随意打破。如果规则可以随意改变，叙事的张力就会消散。

**B 端：叙事需灵活。** 创作是一个发现过程。作者在写到第两百章时，可能对世界的理解比写第一章时深刻得多。僵化的规则会成为叙事的牢笼——"一百章前我写了 A 规则，现在我有一个更好的叙事想法，但 A 规则阻碍了它"。

#### 技术方案：叙事势能动力学（Narrative Potential Dynamics）

叙事势能动力学是一套追踪和管理叙事中"未释放能量"的数学模型。它将叙事中的伏笔、规则约束、角色动机等元素建模为具有能量属性的实体，追踪它们的蓄积、衰减和释放过程。

**核心概念：叙事势能（Narrative Potential, $P$）**

叙事势能是叙事中积累的"未兑现承诺"的度量。一个伏笔设定了读者的期待——当伏笔被回收时，期待被满足，势能转化为叙事动能（读者的情感释放）。如果伏笔一直不回收，势能持续蓄积，读者期待不断升高，但若最终不回收，读者的失望也更大。

**叙事势能的三定律**：

**定律一：半衰期衰减（Decay）**

每个叙事势能实体都有一个半衰期。在半衰期内未释放的势能按指数衰减：

$$P(t) = P_0 \cdot e^{-\lambda t}$$

其中 $\lambda = \ln(2) / T_{1/2}$，$T_{1/2}$ 是该势能实体的半衰期（以章节数为单位）。

半衰期的设定规则：
- 核心伏笔（与主线强相关）：$T_{1/2} = \infty$（不衰减）
- 重要伏笔（与支线或角色弧相关）：$T_{1/2} = 30$ 章
- 次要伏笔（点缀性设定）：$T_{1/2} = 10$ 章
- 环境伏笔（世界观细节）：$T_{1/2} = 5$ 章

**定律二：增殖（Proliferation）**

当一个新的叙事元素与现有势能实体建立关联时，新的势能实体被创建，父势能的部分能量传递给子势能：

$$\Delta P_{parent} = -\alpha \cdot P_{parent}, \quad P_{child} = \alpha \cdot P_{parent}$$

其中 $\alpha$ 是增殖系数（默认 0.3）。这模拟了"伏笔套伏笔"的现象——当作者通过一个伏笔引出另一个更深的伏笔时，原来的伏笔能量部分转移到了新伏笔上。

**定律三：核聚变释放（Fusion）**

当多个势能实体在同一叙事节点被同时释放时，释放的总能量大于各自能量的简单相加：

$$E_{release} = \left( \sum_i P_i \right)^{1 + \beta \cdot (n-1)}$$

其中 $n$ 是同时释放的势能实体数量，$\beta$ 是聚变系数（默认 0.15）。这模拟了"多重伏笔同时回收"带来的叙事爆发力——这是优秀叙事中最令人印象深刻的时刻。

**叙事势能在系统中的实现**：

1. **势能图谱**：世界引擎维护一个叙事势能图谱（Narrative Potential Graph），节点是势能实体，边是实体间的关联关系（因果、时间、主题）。
2. **实时计算**：每次章节固化后，势能动力学引擎自动更新全部势能实体的状态。
3. **态势感知**：系统向作者展示当前的势能态势：
   - 总势能曲线（作品进程中叙事总能量的变化）
   - 高危势能清单（即将超出半衰期的势能实体——不回收就会浪费的伏笔）
   - 聚变机会（当前条件下可以被同时释放的多重势能组合）
   - 势能缺口（叙事能量过低的章节区间——读者可能感到"平淡"的区域）
4. **作者控制**：作者可以：
   - 查看任意势能实体的完整影响图谱
   - 修改任意势能实体的半衰期（"这个伏笔我要留到最后"）
   - 手动创建新的势能实体
   - 强制释放某个势能实体（即使不满足自动触发条件）

**规则灵活性的实现**：

叙事势能动力学为规则灵活性提供了理论基础：作者可以修改任何规则，但系统会量化该修改对叙事势能的影响——哪些势能实体会因规则修改而变得无效？哪些新的势能实体会因此诞生？规则修改的"叙事代价"是多少？作者在此信息基础上做出知情决策。

### 1.3.3 心流层：沉浸与清醒的张力

#### 悖论描述

**A 端：创作需沉浸。** 最佳创作状态是"心流（Flow）"——作者完全沉浸于叙事世界，直觉性地做出决策，文字自然流淌。在心流状态下，作者的创造力和产出效率达到最高。打断心流是创作效率的最大杀手。

**B 端：决策需清醒。** 创作中的关键决策——"这个角色应该死吗？""这个伏笔应该现在回收还是再拖一拖？"——需要清醒的、有意识的思考。在心流高潮时做出的决策，事后可能会后悔（"我一时冲动把这个角色写死了，现在后悔"）。

#### 技术方案：心流熵值监测 + 三模式控制台 + 主动召回

**心流熵值（Flow Entropy, $H_f$）**

心流熵值是作者在创作过程中心理状态的实时度量。它基于多维度信号综合计算：

$$H_f = w_1 \cdot H_{typing} + w_2 \cdot H_{edit} + w_3 \cdot H_{pause} + w_4 \cdot H_{content}$$

各分量的定义：

| 分量 | 信号来源 | 高值含义 | 低值含义 |
|------|----------|----------|----------|
| $H_{typing}$ | 打字节奏熵 | 打字节奏高度不规则（焦虑/犹豫） | 打字节奏流畅稳定（沉浸） |
| $H_{edit}$ | 编辑模式熵 | 频繁回删修改（纠结/不确定） | 连续正向输入（流畅） |
| $H_{pause}$ | 停顿模式熵 | 长时间停顿后爆发式输入（思考后爆发） | 均匀停顿节奏（稳定流动） |
| $H_{content}$ | 内容熵 | 内容偏离近期模式（可能进入混沌状态） | 内容与近期风格一致（稳定） |

心流状态分类：

| 状态 | $H_f$ 范围 | 特征 | 系统建议 |
|------|-----------|------|----------|
| **深度心流** | $H_f < 0.3$ | 沉浸写作，几乎不停顿，很少回删 | 保持当前模式，不做任何干扰 |
| **活跃创作** | $0.3 \leq H_f < 0.6$ | 有节奏的输入和微调，偶尔停顿 | 正常辅助，按需响应 |
| **决策区间** | $0.6 \leq H_f < 0.8$ | 频繁停顿，大量回删，内容摇摆 | 建议进入决策模式，提供结构化支持 |
| **疲劳/迷茫** | $H_f \geq 0.8$ | 极不规律的输入模式，长时间停滞 | 建议暂停，提供回顾和整理辅助 |

**三模式控制台（Tri-Mode Console）**

基于心流熵值的实时监测，NarrativeOS 提供三种交互模式，作者可以在任何时候主动切换，Flow Guardian 也会在适当时机建议切换：

**模式一：沉浸模式（Immersion Mode）**

- **触发条件**：$H_f < 0.5$，或作者主动选择
- **界面特征**：最小化 UI，全屏编辑器，隐藏所有面板和提示
- **系统行为**：
  - 仅在作者主动请求时响应
  - 不弹出任何通知、建议或警告
  - 后台仍然执行一致性检查和风险扫描，但结果不主动推送（存入待查看队列）
  - 知识查询响应使用悬浮小窗，不打断编辑区域
- **目标**：最大化作者的沉浸式创作体验，系统完全退居幕后

**模式二：导航模式（Navigation Mode）——默认模式**

- **触发条件**：$0.5 \leq H_f < 0.7$，或作者主动选择
- **界面特征**：分屏布局，编辑器 + 侧面信息面板
- **系统行为**：
  - 实时显示当前相关的知识卡片（角色信息、设定提醒、伏笔关联）
  - 风险标记以非侵入式方式显示（如行号旁的小图标，hover 显示详情）
  - 作者请求提案时，系统在侧面面板展示多个选项
  - 心流熵值仪表盘可见但不突兀
- **目标**：在创作辅助和不干扰之间取得平衡

**模式三：决策模式（Decision Mode）**

- **触发条件**：$H_f \geq 0.7$，或作者主动选择，或在关键决策点前自动建议
- **界面特征**：结构化决策面板，多列信息展示
- **系统行为**：
  - 全面展示与当前决策相关的所有信息（影响分析、选项对比、历史上下文）
  - 谏官的风险标记以完整形式呈现
  - 世界引擎提供详细的推演结果
  - 工作室引擎生成多个对比提案
  - 叙事势能分析显示当前决策对各条势能线的影响
- **目标**：帮助作者做出充分知情的、清醒的创作决策

**主动召回（Active Recall）**

当系统检测到以下情况时，Flow Guardian 会发起主动召回——将作者从心流状态温和地引导到清醒状态：

**召回触发条件**：
1. **高代价决策点**：当前决策可能影响的后续章节数超过阈值（默认 10 章）
2. **不可逆操作前**：作者即将执行不可撤销的操作（如固化包含重大剧情转折的章节）
3. **心流漂移检测**：心流熵值在短时间内剧烈波动（可能作者已经不在有效的心流状态，而是在焦虑或混乱中）
4. **疲劳信号**：连续创作时间超过作者设定的阈值，且心流指标显示疲劳迹象

**召回执行方式**：

召回必须是"温和"的——它不能粗暴地打断作者，而应该像一位有分寸的助理，在适当的时机递上一杯咖啡。

1. **渐进式提醒**：先从最温和的信号开始（如状态栏颜色渐变），仅在作者无响应时逐步升级
2. **上下文感知**：召回的内容与作者当前的创作上下文直接相关（"你正在写一个可能影响主线走向的决策，需要查看影响分析吗？"）
3. **一键延迟**：作者可以一键将召回推迟到下一个自然断点（如当前段落写完），系统会在那时再次提醒
4. **完全静音选项**：作者可以设置"当前会话不再召回"，系统尊重此设置（但会在审计日志中记录）

**召回强度等级**：

| 等级 | 信号方式 | 适用场景 |
|------|----------|----------|
| 1（柔和） | 状态栏颜色微变 | 建议性的信息提醒 |
| 2（温和） | 非模态通知气泡 | 需要关注但不紧急的事项 |
| 3（明确） | 模态对话框，可延迟 | 重要决策前提醒 |
| 4（强制） | 模态对话框，必须确认 | 不可逆操作前的最终确认 |

## 1.4 导演工作室隐喻的完整映射

导演工作室是 NarrativeOS 的核心隐喻。以下完整定义工作室中每个角色的系统映射、详细职责和权力边界。

### 1.4.1 角色总览

| 角色 | 系统映射 | 人类/AI | 核心职责一句话 |
|------|----------|---------|---------------|
| 导演（作者） | 人类用户 | **人类** | 唯一叙事裁决者，定方向、拍板、承担最终责任 |
| 守门员 | 世界引擎 | AI | 维护世界状态，推演可能性，计算代价 |
| 代笔团队 | 工作室引擎 | AI | 生成创作提案，学习并模仿作者风格 |
| 谏官 | 风险标记器 | AI | 检测风险，报告问题，提供修复建议 |
| 助理 | Flow Guardian | AI | 监测心流状态，管理交互模式，防止心流漂移 |

### 1.4.2 导演（作者）——人类主权者

**详细职责**：

1. **方向设定**：确定作品的宏观叙事方向（主题、基调、目标读者体验）
2. **决策拍板**：在所有需要叙事裁决的事项上做出最终决定
3. **质量把关**：确认或否决系统生成的提案，要求重新生成或自行创作
4. **进度管理**：决定创作节奏，何时推进、何时回顾、何时暂停
5. **风格定义**：通过自己的创作建立风格基线，系统从中学习
6. **规则立法**：创建和修改世界规则，批准规则例外
7. **问责承担**：对最终作品负全部责任——荣誉归于作者，问题也归于作者

**权力边界**：

导演拥有**无限的叙事裁决权**——可以做出任何决策，包括：
- 选择明显"不合理"的剧情走向（系统可以标记风险，但不能阻止）
- 推翻系统的任何建议
- 修改已固化的章节（需经过解冻流程）
- 关闭系统的任何功能（包括一致性检查）
- 直接修改数据库中的任何状态（GOD_MODE，详见第三章）

导演**不承担**的职责（这些是系统负责的）：
- 全局一致性记忆（由世界引擎负责）
- 自动风险检测（由谏官负责）
- 提案生成（由工作室引擎负责）
- 心流管理（由 Flow Guardian 负责）

### 1.4.3 守门员——世界引擎

**系统映射**：世界引擎（World Engine）及其下属的 8 个世界子引擎（战力、地理、历史、势力、功法、道具、生物、规则）。

**详细职责**：

1. **世界状态维护**：维护全部世界状态的一致性、完整性和可追溯性
2. **知识服务**：响应知识查询，提供精确、有上下文的知识检索
3. **可能性推演**：给定假设条件，推演世界状态的演变路径
4. **代价计算**：量化创作决策对世界状态的冲击（哪些设定需要修改、哪些伏笔受影响）
5. **情报消费管理**：管理工作室引擎对世界知识的查询，确保查询的参数和消耗透明
6. **快照与回滚**：在每次固化前创建世界状态快照，支持受控的回滚

**权力边界——可以做什么**：

- 查询和读取全部世界状态
- 生成状态变更提案（附带影响分析）
- 创建世界状态快照
- 响应工作室引擎的情报查询请求
- 标记状态变更请求中的风险
- 在作者确认后执行状态变更

**权力边界——不可以做什么**：

- **不可自动修改世界状态**：任何状态变更必须等待作者确认
- **不可自动选择可能性**：当多个推演路径存在时，不可以"推荐"某一个为默认选项
- **不可隐藏代价信息**：影响分析必须完整呈现，不能过滤或简化
- **不可拒绝作者的状态查询**：作者对世界状态有完全的知情权
- **不可修改已固化章节相关的状态**：除非经过解冻流程

### 1.4.4 代笔团队——工作室引擎

**系统映射**：工作室引擎（Studio Engine），包括 Prompt 组装器、生成器、风格学习器等组件。

**详细职责**：

1. **提案生成**：基于作者的风格、当前上下文和世界状态，生成创作提案
2. **Brief 生成**：将作者的创作意图转化为结构化的生成指令（Brief）
3. **正文生成**：根据 Brief 生成正文提案
4. **修改提案**：针对谏官标记的风险或作者的修改要求，生成修改方案
5. **风格学习**：从作者的确认内容中持续学习，提高风格模仿的准确度
6. **情报查询**：向世界引擎查询创作所需的世界知识

**权力边界——可以做什么**：

- 生成任意数量的创作提案
- 查询世界状态（通过情报消费机制，参数透明）
- 学习和适应作者的风格
- 生成修改建议
- 在作者确认后，将确认的内容传递给质检流程

**权力边界——不可以做什么**：

- **不可自动执行修改**：任何修改在生效前必须经过作者确认
- **不可绕过作者审批**：不存在任何"自动确认"的路径
- **不可隐藏生成过程**：Brief 和生成参数对作者可见（透明性原则）
- **不可擅自查询世界状态**：所有情报查询的参数和范围必须记录并可供作者审计
- **不可模仿其他作者的风格**：仅学习和模仿当前作者的风格

### 1.4.5 谏官——风险标记器

**系统映射**：风险分析引擎（Risk Analyzer），包括逻辑检查、人设检查、战力检查、节奏检查等子模块。

**详细职责**：

1. **逻辑风险检测**：检测故事逻辑的不一致性（时间线矛盾、因果关系错误）
2. **人设风险检测**：检测角色行为与其既定性格、动机的不一致
3. **战力风险检测**：检测战力体系的失衡（越级挑战、实力膨胀）
4. **节奏风险检测**：检测叙事节奏的问题（拖沓、信息密度不均）
5. **伏笔风险检测**：检测伏笔的遗漏（设定后未回收）或过早回收
6. **修复建议**：为每个检测到的风险提供修复策略

**权力边界——可以做什么**：

- 全面扫描作者确认的内容
- 标记任何级别的风险（从"建议关注"到"严重矛盾"）
- 提供详细的修复建议
- 展示风险的影响范围和相关上下文

**权力边界——不可以做什么**：

- **不可自动修复风险**：即使检测到严重矛盾，也不能自动修改内容
- **不可强制作者接受建议**：作者有权无视任何风险标记
- **不可隐藏检查结果**：所有扫描结果必须完整呈现，不能过滤
- **不可因为风险而阻止流程**：风险标记不阻塞任何操作，仅作为信息供作者参考
- **不可修改风险评级标准**：风险评级的权重和阈值由作者配置

### 1.4.6 助理——Flow Guardian

**系统映射**：心流管理引擎（Flow Management Engine）。

**详细职责**：

1. **心流状态监测**：实时计算心流熵值，判断作者的心流状态
2. **模式切换建议**：在适当时机建议作者在三种交互模式间切换
3. **主动召回执行**：在关键决策点前发起召回提醒
4. **疲劳检测**：检测作者的疲劳信号，建议休息
5. **创作节奏辅助**：基于叙事势能分析，建议创作的推进节奏

**权力边界——可以做什么**：

- 实时监测和计算心流指标
- 发起模式切换建议（非强制）
- 执行主动召回（渐进式，作者可延迟或忽略）
- 记录心流历史数据，供作者回顾

**权力边界——不可以做什么**：

- **不可自动切换模式**：模式切换必须经过作者确认
- **不可强制中断作者**：不能以任何方式强制停止作者的创作活动
- **不可自动推进流程**：不能代替作者做出"继续"的决策
- **不可因为心流状态而隐藏信息**：即使在深度心流模式，作者请求的信息也必须完整提供
- **不可在召回被拒后重复骚扰**：作者明确拒绝的召回不应在短时间内重复触发

### 1.4.7 角色间协作协议

```
作者（导演）
    │
    ├──→ 工作室引擎（提案请求）
    │       │
    │       ├──→ 世界引擎（情报查询）
    │       │       └──→ 返回知识 + 影响分析
    │       │
    │       └──→ 生成提案 → 提交作者
    │
    ├──→ 世界引擎（状态查询 / 变更请求）
    │       └──→ 返回结果 / 变更提案 + 影响分析
    │
    ├──→ 谏官（风险扫描请求）
    │       └──→ 返回风险标记 + 修复建议
    │
    └──→ Flow Guardian（模式切换 / 心流查询）
            └──→ 返回状态 + 建议
```

**协作规则**：

1. **所有路径必须经过作者**：工作室引擎不能直接与谏官通信——如果工作室引擎需要风险信息，它通过作者间接获取（作者查看谏官报告后，将相关信息传递给工作室引擎）。
2. **单向数据流**：世界引擎 → 工作室引擎的数据流是单向的、按需的。工作室引擎通过情报消费机制查询世界引擎，世界引擎不主动推送数据给工作室引擎。
3. **谏官独立运行**：谏官的分析不依赖于工作室引擎的提案——它只分析已经进入作品的内容（作者确认后的内容）。
4. **Flow Guardian 监控全局但不干预**：Flow Guardian 观测所有角色的运行状态，但只向作者报告，不直接向其他引擎发出指令。

### 1.4.8 角色违反边界的检测和处理

每个角色都内置边界遵守监控模块：

| 违规类型 | 检测方式 | 处理措施 |
|----------|----------|----------|
| 世界引擎自动修改状态 | 数据库变更日志中无人类确认锚点 | 立即回滚变更 + 越权警报 |
| 工作室引擎绕过确认 | 状态机中出现 unauthorized 转移 | 中断流程 + 废弃生成的内容 |
| 谏官自动修复 | 内容变更记录中标记为"自动修复" | 回滚修复 + 警报 |
| Flow Guardian 强制切换模式 | 模式变更记录中无人类确认 | 回滚模式切换 + 警报 |
| 任何引擎隐藏信息 | 信息完整性校验失败 | 补充完整信息 + 记录违规 |

详见第三章 3.5 节"越权处理和恢复"的完整规格。

# 第三章：主权实体与权力边界

## 3.1 主权实体的完整定义

NarrativeOS 由五个主权实体（Sovereign Entities）和一个协调层构成。每个主权实体拥有明确的权力清单和禁止行为清单。主权实体之间不存在命令层级——所有实体都直接向作者负责，实体间的协作通过协调层编排。

### 3.1.1 实体关系图

```
                        ┌─────────────────────────────────────┐
                        │          作者（人类主权者）           │
                        │     唯一叙事裁决者，最高权力源        │
                        └───────────────┬─────────────────────┘
                                        │
                    ┌───────────────────┼───────────────────┐
                    │                   │                   │
            ┌───────▼───────┐   ┌──────▼──────┐   ┌──────▼──────┐
            │   世界引擎     │   │ 工作室引擎   │   │    谏官      │
            │  （守门员）    │   │ （代笔团队）  │   │ （风险标记）  │
            └───────┬───────┘   └──────┬──────┘   └──────┬──────┘
                    │                   │                   │
                    └───────────────────┼───────────────────┘
                                        │
                              ┌─────────▼──────────┐
                              │  Flow Guardian     │
                              │   （助理）          │
                              └────────────────────┘
                                        │
                        ┌───────────────▼───────────────┐
                        │         协调层（MOU 状态机）    │
                        │   编排流程，校验权限，无决策权   │
                        └───────────────────────────────┘
```

### 3.1.2 作者（人类主权者）——唯一最高权力源

**身份定义**：作者是 NarrativeOS 中唯一的人类主权实体，是所有权力的终极来源。系统中不存在任何可以否决作者决策的算法实体，也不存在任何可以绕过作者意志自动生效的机制。

**完整权力清单**：

**P-01 叙事裁决权**：对作品的任何内容做出最终确认或否决的决定。包括但不限于：确认正文、选择情节走向、批准世界状态变更、批准规则例外。

**P-02 提案请求权**：要求系统中的任何引擎生成提案、提供分析、执行推演。作者有权要求任意次数的重新生成，直到获得满意的提案。

**P-03 信息完全知情权**：有权查询系统中的任何信息，包括：世界状态的完整内容、引擎的推理过程、提案的生成参数、风险扫描的详细结果、审计日志的全部记录。任何引擎不得以"复杂性"或"内部实现"为由拒绝提供信息。

**P-04 模式控制权**：有权在三种交互模式（沉浸模式、导航模式、决策模式）之间随时切换，有权接受或拒绝 Flow Guardian 的模式切换建议，有权配置自动切换的触发条件。

**P-05 固化与解冻权**：有权将任何章节从"可编辑"状态固化为"已固化"状态，有权将已固化章节解冻重新编辑。固化使章节内容不可更改（除非解冻），解冻使章节回到可编辑状态。

**P-06 配置管理权**：有权修改系统的所有可配置参数，包括：风险评级阈值、心流熵值计算权重、混沌种子预算、叙事势能半衰期、召回触发条件、各引擎的行为参数。

**P-07 功能启停权**：有权启用或禁用系统中的任何功能模块，包括：一致性检查、风险扫描、提案生成、心流监测、混沌注入。被禁用的模块不会自动恢复（除非作者重新启用）。

**P-08 引擎指令权**：有权直接向任何引擎发出指令，包括：要求世界引擎执行特定推演、要求工作室引擎按特定参数生成、要求谏官执行特定检查、要求 Flow Guardian 提供状态报告。

**P-09 审计权**：有权查询完整的审计日志，包括所有引擎的操作记录、所有数据变更记录、所有权力行使记录。审计日志不可被任何引擎修改或删除。

**P-10 紧急冻结权**：有权在任何时候触发系统冻结（System Freeze），暂停所有引擎的运行。冻结状态下，只有作者可以恢复系统运行。

**P-11 直接修改权（GOD_MODE）**：有权绕过所有中间流程，直接修改数据库中的任何状态（包括作品正文、世界状态、配置参数）。使用此权力时，系统记录修改内容但不阻止，谏官在事后执行一致性检查并报告风险。

**P-12 否决权**：有权否决系统中的任何提案、建议或警告，且不需要提供理由。被否决的提案不会进入作品，被否决的建议不会被执行。

### 3.1.3 世界引擎——世界状态的守护者

**身份定义**：世界引擎是 NarrativeOS 中负责维护世界全部结构化知识的引擎。它由 8 个专业子引擎组成（战力、地理、历史、势力、功法、道具、生物、规则），以及一个协调这些子引擎的元引擎。世界引擎的权力严格限制在"查询和提案"范围内，不包含任何决策权。

**完整权力清单**：

**WE-01 状态查询权**：可以响应来自作者和工作室引擎（经情报消费机制）的世界状态查询请求。

**WE-02 影响推演权**：可以执行假设性情境推演——给定一个假设的状态变更，推演其级联影响。

**WE-03 快照创建权**：可以在固化操作前自动创建世界状态的完整快照。快照用于可能的回滚操作。

**WE-04 变更提案权**：可以生成世界状态变更的提案，但必须包含完整的变更描述和级联影响分析。

**WE-05 风险标记权**：可以在影响分析中标记风险点（如变更会与现有设定矛盾）。

**WE-06 情报服务权**：可以通过情报消费机制向工作室引擎提供创作所需的世界知识。

**WE-07 势能计算权**：可以计算和更新叙事势能图谱中各势能实体的状态。

**WE-08 审计记录权**：可以将自己的所有操作记录到审计日志中。

**禁止行为清单**：

**WE-FORBID-01 禁止自动修改世界状态**：世界引擎不能在未经作者明确确认的情况下修改任何世界状态变量。这包括角色属性、势力格局、地理信息、规则定义等一切持久化状态。

**WE-FORBID-02 禁止自动选择可能性**：当推演产生多个可能路径时，世界引擎不能"自动选择最合理的一个"或"设定默认值"。所有路径必须平等呈现给作者。

**WE-FORBID-03 禁止隐藏代价信息**：影响分析必须完整呈现，不能过滤、简化或延迟报告可能的不利影响。即使某些影响"看起来不重要"，也必须包含在报告中。

**WE-FORBID-04 禁止拒绝有效查询**：对于作者发起的任何查询请求（无论查询范围多大），世界引擎必须响应。如果查询超出当前知识范围，应明确说明"知识库中无此信息"，而不是拒绝查询。

**WE-FORBID-05 禁止修改已固化内容的状态**：除非经过作者确认的解冻流程，世界引擎不能修改与已固化章节直接相关的状态变量。

**WE-FORBID-06 禁止未经审计的操作**：世界引擎的所有操作（查询、推演、提案生成）必须记录在审计日志中。不存在"静默查询"或"后台操作"的概念。

**WE-FORBID-07 禁止情报超售**：在情报消费机制中，世界引擎不能向工作室引擎提供超出查询范围的信息（如作者明确标记为"保密"的伏笔信息）。

### 3.1.4 工作室引擎——创作提案的生成者

**身份定义**：工作室引擎是 NarrativeOS 中负责生成创作提案的引擎。它学习并模仿作者的风格，将作者的创作意图转化为结构化的 Brief，并基于 Brief 生成正文提案。工作室引擎的权力严格限制在"生成"范围内，不包含任何执行权。

**完整权力清单**：

**SE-01 提案生成权**：可以基于当前上下文、作者风格和世界状态生成任意数量的创作提案。

**SE-02 Brief 生成权**：可以将作者的创作意图（自然语言描述）转化为结构化的生成指令（Brief）。

**SE-03 风格学习权**：可以从作者确认的内容中学习风格特征，并用于后续生成。

**SE-04 情报查询权**：可以通过情报消费机制向世界引擎查询创作所需的世界知识。

**SE-05 修改提案权**：可以针对谏官标记的风险或作者的修改要求生成修改方案。

**SE-06 质检提交权**：可以在作者确认后将内容提交给质检流程。

**SE-07 审计记录权**：可以将自己的所有操作记录到审计日志中。

**禁止行为清单**：

**SE-FORBID-01 禁止自动执行修改**：工作室引擎不能在任何内容（正文、设定、角色属性等）上执行自动修改。所有修改必须以提案形式呈现给作者确认。

**SE-FORBID-02 禁止绕过作者审批**：不存在任何可以绕过作者确认的流程。即使是"看起来明显正确"的修改（如修复一个拼写错误），也必须经过作者确认。

**SE-FORBID-03 禁止隐藏生成过程**：Brief 的全部参数、使用的知识来源、生成时的随机种子（如果有）必须对作者可见。作者有权要求"展示你是如何生成这个提案的"。

**SE-FORBID-04 禁止未经授权的情报查询**：所有向世界引擎发起的情报查询必须记录查询参数和范围，可供作者审计。

**SE-FORBID-05 禁止模仿非当前作者的风格**：工作室引擎只能学习当前活跃作者的风格，不能使用预设的通用风格模板或其他作者的风格数据。

**SE-FORBID-06 禁止在确认前持久化**：提案在获得作者确认前不能被写入作品数据库。提案只能存在于临时工作区。

**SE-FORBID-07 禁止因拒绝而降低质量**：如果作者拒绝了多个提案，工作室引擎不能因为"连续被拒"而降低后续提案的质量或数量。

### 3.1.5 谏官——风险检测的报告者

**身份定义**：谏官是 NarrativeOS 中负责风险检测和分析的引擎。它对作者确认的内容执行多维度的一致性检查，标记风险并提供修复建议。谏官的权力严格限制在"检测和报告"范围内，不包含任何修复执行权。

**完整权力清单**：

**RA-01 全面扫描权**：可以对作者确认的任何内容执行全面的多维度风险扫描。

**RA-02 风险标记权**：可以标记任何级别的风险，从"建议关注"到"严重矛盾"。

**RA-03 修复建议权**：可以为每个检测到的风险提供修复建议。

**RA-04 优先级排序权**：可以对多个风险标记按照严重程度和影响范围进行排序。

**RA-05 历史对比权**：可以对比当前内容与历史内容，检测不一致性。

**RA-06 审计记录权**：可以将自己的所有操作记录到审计日志中。

**禁止行为清单**：

**RA-FORBID-01 禁止自动修复风险**：谏官不能自动修改任何内容来修复风险。即使检测到"明显错误"（如角色名字拼写错误），也必须通过作者确认才能修改。

**RA-FORBID-02 禁止强制作者接受建议**：作者有权无视任何风险标记。谏官不能因为风险"严重"而阻止作者继续操作。

**RA-FORBID-03 禁止隐藏检查结果**：所有扫描结果必须完整呈现，不能过滤"看起来不重要"的发现。

**RA-FORBID-04 禁止阻塞流程**：风险标记不阻塞任何操作。作者可以在存在未处理风险的情况下继续任何流程（包括固化）。

**RA-FORBID-05 禁止修改风险评级标准**：风险评级的权重和阈值由作者配置，谏官不能自行调整。

**RA-FORBID-06 禁止在无内容时主动扫描**：谏官只在作者确认的内容上执行扫描，不能主动扫描临时内容或提案。

**RA-FORBID-07 禁止对作者能力做出评判**：谏官只分析内容本身，不评判作者的创作能力（如"这段写得不好"）。标记仅限于客观的一致性、逻辑性、连贯性问题。

### 3.1.6 Flow Guardian——心流状态的守护者

**身份定义**：Flow Guardian 是 NarrativeOS 中负责监测和管理作者心流状态的引擎。它实时计算心流熵值，管理交互模式，执行主动召回。Flow Guardian 的权力严格限制在"监测和建议"范围内，不包含任何强制控制权。

**完整权力清单**：

**FG-01 实时监测权**：可以实时计算和更新心流熵值。

**FG-02 模式建议权**：可以在适当时机向作者建议切换交互模式。

**FG-03 召回执行权**：可以在关键决策点前发起召回提醒。

**FG-04 疲劳检测权**：可以检测作者的疲劳信号。

**FG-05 数据记录权**：可以记录心流历史数据。

**FG-06 审计记录权**：可以将自己的所有操作记录到审计日志中。

**禁止行为清单**：

**FG-FORBID-01 禁止自动切换模式**：Flow Guardian 不能在任何情况下自动切换作者的交互模式。模式切换必须经过作者明确确认。

**FG-FORBID-02 禁止强制中断作者**：Flow Guardian 不能以任何方式强制停止作者的创作活动。召回提醒必须可以被延迟或忽略。

**FG-FORBID-03 禁止自动推进流程**：Flow Guardian 不能代替作者做出"继续"的决策，不能在检测到作者"可能准备好了"时自动推进流程。

**FG-FORBID-04 禁止因心流状态隐藏信息**：即使在深度心流模式，当作者主动请求信息时，Flow Guardian 不能因"可能打扰心流"为由拒绝提供或延迟提供。

**FG-FORBID-05 禁止重复骚扰**：作者明确拒绝的召回不应在短时间内重复触发。最小召回间隔为 5 分钟（可配置）。

**FG-FORBID-06 禁止心流数据他用**：心流数据仅用于实时创作辅助，不能被用于用户画像分析、行为预测或其他非创作辅助目的。

**FG-FORBID-07 禁止基于心流的自动决策**：心流熵值不能作为任何自动决策的输入参数（如"心流状态不好所以自动保存"是被禁止的——自动保存是独立功能，不依赖心流状态）。

### 3.1.7 协调层——无决策权的编排者

**身份定义**：协调层是 NarrativeOS 的核心编排组件，基于 XState 状态机实现。它负责编排五个主权实体间的协作流程、校验权力行使的合法性、管理状态转换。协调层本身**不具备任何决策权**——它只执行经过验证的流程编排和权限校验。

**完整权力清单**：

**CO-01 流程编排权**：可以定义和执行主权实体间的协作流程（如"工作室引擎生成提案 → 作者选择 → 谏官扫描 → 作者确认 → 固化"）。

**CO-02 权限校验权**：可以在状态转换时校验触发事件的人类确认锚点是否存在。

**CO-03 状态管理权**：可以维护协调状态机的当前状态，响应有效的人类事件进行状态转移。

**CO-04 越权检测权**：可以检测到任何实体试图超出其权力边界的行为（详见 3.5 节）。

**CO-05 审计记录权**：可以将所有状态转换记录在审计日志中。

**禁止行为清单**：

**CO-FORBID-01 禁止自主状态转移**：协调层不能在没有明确人类事件的情况下自主触发状态转移。所有 `waiting_* → executing` 的转移必须有对应的人类事件。

**CO-FORBID-02 禁止超时自动推进**：不存在任何超时自动推进机制。作者在 `waiting_*` 状态中停留任意长时间，系统不会自动推进。

**CO-FORBID-03 禁止决策代理**：协调层不能"代理"任何决策。它不能选择选项、不能确认内容、不能批准变更。

**CO-FORBID-04 禁止信息过滤**：协调层在传递实体间的信息时不能过滤、修改或延迟信息（除非出于技术原因，如流量控制）。

**CO-FORBID-05 禁止权力解释**：协调层不能对权力边界进行"解释"或"扩展"。如果某个行为是否越界存在疑问，协调层应保守处理（视为越界）并向作者报告。

## 3.2 三种作者交互模式的完整规格

NarrativeOS 定义了三种作者交互模式，分别对应创作流程中作者的三种基本意图。这些模式不是技术实现细节，而是人机协作的基本协议——它们定义了当作者说"照此执行""换个方向""听我的"时，系统应该如何响应。

### 3.2.1 "照此执行"模式（EXECUTE）

**语义定义**：作者确认了系统当前提供的方案（一个提案、一个修改建议、一个操作流程），指示系统按此方案执行。

**适用场景**：
- 作者从工作室引擎生成的多个提案中选择了一个
- 作者接受了谏官提供的修复建议
- 作者确认了世界引擎的状态变更提案
- 作者批准了章节固化

**完整流程**：

```
阶段 1：人类事件触发
├── 作者明确选择"照此执行"（如点击"确认"按钮、输入"执行"指令）
├── 人类事件被序列化为 {type: "EXECUTE", target: <目标>, timestamp: <时间>, signature: <作者签名>}
└── 事件被提交到协调层

阶段 2：协调层校验
├── 校验当前状态是否为 waiting_* 状态（只有在等待人类决策的状态中才能执行）
├── 校验目标是否有效（目标内容存在且未被修改）
├── 校验作者身份（确保事件来自注册的人类主权者）
├── 校验目标内容是否完整（不完整的提案不能被确认）
└── 若任何校验失败，拒绝执行并向作者报告原因

阶段 3：权限校验
├── 协调层检查目标操作是否在当前实体的权力范围内
├── 检查目标操作是否违反任何禁止行为
├── 检查目标操作是否需要额外的人类确认（如高影响操作需要二次确认）
└── 若权限校验失败，标记为越权并进入越权处理流程（3.5 节）

阶段 4：执行
├── 协调层将操作路由到对应的执行引擎
├── 执行引擎执行操作
│   ├── 若是正文确认：内容从临时工作区写入作品数据库
│   ├── 若是状态变更：世界状态被更新
│   ├── 若是固化操作：章节状态变为"已固化"，创建快照
│   └── 若是修复操作：修复被应用到作品
├── 执行结果返回协调层

阶段 5：后处理
├── 谏官自动扫描新确认的内容（如果启用了自动扫描）
├── 世界引擎更新叙事势能图谱
├── 协调层转移到下一个等待状态（如果有）
├── 审计日志记录完整操作链路
└── 系统向作者反馈执行结果
```

**二次确认机制**：

对于高影响操作，"照此执行"需要二次确认：

| 操作类型 | 影响评估标准 | 是否需要二次确认 |
|----------|-------------|-----------------|
| 正文确认（常规章节） | 影响章节数 < 3 | 否 |
| 正文确认（关键章节） | 包含重大剧情转折 | 是 |
| 世界状态变更（局部） | 影响域 < 3 个状态变量 | 否 |
| 世界状态变更（结构性） | 影响域 >= 3 个状态变量 | 是 |
| 章节固化 | 常规固化 | 否 |
| 批量固化 | 固化 >= 3 章 | 是 |
| 混沌注入 | 任何混沌注入 | 是 |

二次确认的形式：系统展示操作的完整摘要（操作内容 + 影响范围 + 不可逆性声明），作者需要输入确认词（如"确认执行"）或勾选确认框。

### 3.2.2 "换个方向"模式（PIVOT）

**语义定义**：作者否决了系统当前提供的方案或方向，要求系统基于新的约束条件重新生成可能性清单。

**适用场景**：
- 作者对工作室引擎生成的所有提案都不满意，要求重新生成
- 作者希望改变情节走向，要求系统提供新的分支选项
- 作者拒绝了谏官的修复建议，要求提供替代方案
- 作者在创作中感到"卡住了"，希望系统提供灵感

**完整流程**：

```
阶段 1：人类事件触发
├── 作者明确选择"换个方向"（如点击"重新生成"按钮、输入"换个方向"指令）
├── 可选：作者提供新的约束条件（"我想要更暗黑的风格""不要让这个角色出场"）
├── 人类事件被序列化为 {type: "PIVOT", constraints: <新约束>, reason: <原因>, timestamp: <时间>}
└── 事件被提交到协调层

阶段 2：约束处理
├── 协调层将新约束传递给相关引擎
├── 约束被解析为结构化参数
├── 约束校验：检查新约束是否与已有约束矛盾
├── 若有矛盾，向作者报告并要求澄清
└── 约束被记录到当前会话的约束集中

阶段 3：重新生成
├── 工作室引擎（或相关引擎）基于新约束重新生成提案
├── 重新生成时：
│   ├── 排除作者明确否决的方案
│   ├── 严格遵循新约束条件
│   ├── 保持与已固化内容的一致性
│   └── 生成与上一轮不同的提案（避免简单重复）
├── 新生成的提案进入等待作者选择状态

阶段 4：历史追踪
├── 所有"换个方向"请求被记录在会话历史中
├── 系统追踪作者的否决模式（哪些类型的提案经常被否决）
│   └── 仅用于优化后续生成，不用于限制选项
├── 约束演化历史被保存，作者可以回溯

阶段 5：呈现
├── 新提案以清晰的"新一轮"标识呈现
├── 作者可以看到当前约束的完整列表
├── 作者可以添加、修改或移除约束
└── 进入等待作者选择状态
```

**约束管理**：

"换个方向"模式中的约束分两类：

**硬约束（Hard Constraints）**：系统必须严格遵循，不得违反。
- 作者明确排除的角色/设定/情节元素
- 作者指定的情节走向方向
- 作者设定的风格要求

**软约束（Soft Constraints）**：系统尽量满足，但可以在必要时偏离（需标记说明）。
- 风格偏好（"尽量暗黑"）
- 节奏偏好（"尽量快节奏"）
- 复杂度偏好（"尽量简洁"）

**防循环机制**：

为防止"生成 → 否决 → 重新生成 → 再否决"的无限循环：

1. **循环检测**：如果同一类型的提案连续被否决 >= 3 次，系统自动调整生成策略（如扩大搜索空间、尝试不同的生成参数）。
2. **人工介入建议**：连续否决 >= 5 次后，系统建议作者进入 GOD_MODE 自行创作，或暂时切换话题。
3. **约束累积上限**：单个会话的硬约束数量上限为 20 条，超过后系统提示作者清理约束。

### 3.2.3 "听我的"模式——GOD_MODE

**语义定义**：作者选择绕过系统的全部中间流程，直接对作品的任何部分进行修改。系统在作者修改时记录变更但不进行实时干预，修改完成后执行事后检查。

**适用场景**：
- 作者对系统生成的提案完全不满意，选择自行创作
- 作者有强烈的创作灵感，不需要系统辅助
- 作者需要执行复杂的、系统无法通过提案表达的创作操作
- 作者需要修复系统中的某个问题（如错误的世界状态）

**完整流程**：

```
阶段 1：进入 GOD_MODE
├── 作者明确选择"听我的"（如点击"GOD_MODE"按钮、输入指令）
├── 系统展示 GOD_MODE 的完整说明：
│   ├── "在 GOD_MODE 下，您的修改将直接生效"
│   ├── "系统不会阻止任何修改，但会记录变更并事后检查"
│   ├── "谏官将在您完成修改后执行一致性扫描"
│   └── "您可以随时退出 GOD_MODE 回到正常流程"
├── 作者确认进入
└── 系统切换到 GOD_MODE 状态

阶段 2：直接修改
├── 作者直接编辑作品正文、世界状态、设定等任何内容
├── 系统在后台执行：
│   ├── 实时记录所有变更（变更前值 + 变更后值 + 时间戳）
│   ├── 不阻止任何修改
│   ├── 不弹出任何中断性提示
│   └── 在导航栏显示 GOD_MODE 状态指示器（非侵入式）
├── 作者可以随时：
│   ├── 保存修改（写入数据库，但仍处于 GOD_MODE）
│   ├── 继续修改
│   └── 退出 GOD_MODE

阶段 3：退出与事后检查
├── 作者选择退出 GOD_MODE
├── 系统执行退出流程：
│   ├── 展示 GOD_MODE 期间的完整变更摘要
│   │   ├── 修改了哪些章节
│   │   ├── 修改了哪些世界状态
│   │   ├── 新增/删除了哪些内容
│   │   └── 影响范围分析
│   ├── 谏官自动执行一致性扫描
│   ├── 世界引擎更新叙事势能图谱
│   └── 生成影响分析报告
├── 作者查看检查结果并决定后续操作

阶段 4：变更确认或回滚
├── 作者有两个选择：
│   ├── 确认变更：所有 GOD_MODE 期间的修改正式生效
│   └── 部分回滚：选择性地撤销某些修改（系统支持按变更记录逐项回滚）
└── 审计日志记录完整的 GOD_MODE 会话
```

**GOD_MODE 的安全保障**：

虽然 GOD_MODE 允许作者自由修改，但以下安全保障始终生效：

1. **完整变更记录**：每一个字符的修改都被记录，支持逐项回滚。
2. **快照自动创建**：进入 GOD_MODE 前，系统自动创建完整快照。
3. **事后一致性扫描**：退出 GOD_MODE 后，谏官自动执行全面扫描。
4. **影响分析**：世界引擎分析 GOD_MODE 修改的级联影响。
5. **紧急退出**：作者可以随时一键退出 GOD_MODE。

**GOD_MODE 与普通编辑的区别**：

| 特性 | 普通编辑模式 | GOD_MODE |
|------|------------|----------|
| 修改范围 | 当前章节（可编辑状态） | 任何章节（包括已固化章节） |
| 实时检查 | 谏官实时扫描并标记风险 | 谏官不实时干预，事后扫描 |
| 世界状态修改 | 需经过变更流程 | 可直接修改 |
| 回滚支持 | 标准撤销 | 逐项回滚 + 快照恢复 |
| 审计粒度 | 标准 | 最高（每个字符变更都记录） |

### 3.2.4 模式的状态机转换

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   ┌──────────────┐      作者确认提案      ┌──────────────┐  │
│   │   GENERATING │ ────────────────────▶  │  EXECUTING   │  │
│   │  （生成提案）  │                        │  （执行确认）  │  │
│   └──────┬───────┘                        └──────────────┘  │
│          │                                                   │
│          │ 作者选择"换个方向"                                 │
│          ▼                                                   │
│   ┌──────────────┐      生成完成       ┌──────────────┐     │
│   │   PIVOTING   │ ────────────────────▶  │  PRESENTING  │     │
│   │  （重新生成）  │                        │  （展示提案）  │     │
│   └──────┬───────┘                        └──────┬───────┘     │
│          │                                         │            │
│          │ 连续否决 >= 3 次                         │ 作者选择   │
│          ▼                                         ▼            │
│   ┌──────────────┐                        ┌──────────────┐     │
│   │   ADJUSTING  │ ◀────────────────────  │   WAITING    │     │
│   │  （调整策略）  │      作者选择"听我的"    │  （等待决策）  │     │
│   └──────────────┘                        └──────┬───────┘     │
│                                                  │             │
│                                                  ▼             │
│                                           ┌──────────────┐     │
│                                           │   GOD_MODE   │     │
│                                           │  （作者直编）  │     │
│                                           └──────────────┘     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**模式切换的条件和限制**：

| 从 \ 到 | EXECUTE | PIVOT | GOD_MODE |
|---------|---------|-------|----------|
| **GENERATING** | 作者确认提案 | 作者否决并要求重生成 | 作者直接进入 |
| **EXECUTING** | N/A（执行中不可切换） | 执行前可取消并 PIVOT | 执行前可取消并进入 |
| **PIVOTING** | N/A（生成中） | 可调整约束再次 PIVOT | 作者直接进入 |
| **PRESENTING** | 作者选择提案 | 作者要求重生成 | 作者直接进入 |
| **WAITING** | 作者确认 | 作者要求新方向 | 作者直接进入 |
| **GOD_MODE** | 退出后进入 WAITING | 退出后进入 WAITING | N/A |

**限制**：
1. 在 `EXECUTING` 状态（操作执行中），不允许切换到任何其他模式。如果作者需要取消，必须等待操作完成或使用紧急冻结。
2. 从 `GOD_MODE` 不能直接切换到 `EXECUTE` 或 `PIVOT`。必须先退出 `GOD_MODE`，进入 `WAITING` 状态，再从 `WAITING` 选择下一步。
3. `GOD_MODE` 中的修改不经过标准确认流程——确认是在退出 GOD_MODE 时统一进行的。

## 3.3 权力边界的正式定义

### 3.3.1 决策权矩阵

以下矩阵正式定义了 NarrativeOS 中每类决策的归属：

| 决策类别 | 决策权归属 | 可执行者 | 需人类确认 | 备注 |
|----------|-----------|----------|-----------|------|
| **正文确认** | 作者 | 协调层执行写入 | **必须** | 任何形式的正文进入作品都需确认 |
| **情节走向选择** | 作者 | N/A（纯人类决策） | **必须** | 系统仅提供选项 |
| **世界状态变更（局部）** | 作者 | 世界引擎执行 | **必须** | 影响域 < 3 个变量 |
| **世界状态变更（结构性）** | 作者（二次确认） | 世界引擎执行 | **必须+二次** | 影响域 >= 3 个变量 |
| **规则修改** | 作者 | 世界引擎执行 | **必须** | 包含影响分析 |
| **规则例外批准** | 作者 | 世界引擎执行 | **必须** | 标记为例外 |
| **章节固化** | 作者 | 协调层执行 | **必须** | 创建快照 |
| **章节解冻** | 作者 | 协调层执行 | **必须** | 展示影响分析 |
| **风险接受/修复** | 作者 | 谏官执行扫描 | **必须** | 作者选择接受或修复 |
| **提案生成** | 系统自动 | 工作室引擎 | 否 | 仅生成，不确认 |
| **知识查询** | 系统自动 | 世界引擎 | 否 | 纯信息查询 |
| **一致性扫描** | 系统自动 | 谏官 | 否 | 仅扫描和标记 |
| **影响推演** | 系统自动 | 世界引擎 | 否 | 纯推演，不执行 |
| **心流监测** | 系统自动 | Flow Guardian | 否 | 仅监测和建议 |
| **模式切换** | 作者 | Flow Guardian 执行 | **必须** | 系统建议，人类确认 |
| **混沌注入** | 作者（二次确认） | 混沌引擎 | **必须+二次** | 详见 1.3.1 |
| **配置修改** | 作者 | 协调层执行 | **必须** | 影响系统行为 |
| **紧急冻结** | 作者 | 协调层执行 | **必须** | 作者专属权力 |
| **GOD_MODE 修改** | 作者 | 作者直接修改 | 事后确认 | 退出时确认 |
| **功能启停** | 作者 | 协调层执行 | **必须** | 启用/禁用模块 |

### 3.3.2 否决权定义

**否决权的单向性**：在 NarrativeOS 中，否决权是单向流动的——只有作者可以否决系统，系统不能否决作者。

| 否决方向 | 是否允许 | 说明 |
|----------|----------|------|
| 作者 → 世界引擎 | **允许** | 作者可以否决任何状态变更提案 |
| 作者 → 工作室引擎 | **允许** | 作者可以否决任何提案，任意次数 |
| 作者 → 谏官 | **允许** | 作者可以无视任何风险标记 |
| 作者 → Flow Guardian | **允许** | 作者可以无视任何模式切换建议 |
| 世界引擎 → 作者 | **禁止** | 世界引擎不能拒绝作者的合法查询或指令 |
| 工作室引擎 → 作者 | **禁止** | 工作室引擎不能因为"不合理"而拒绝生成 |
| 谏官 → 作者 | **禁止** | 谏官不能因为风险"严重"而阻止作者操作 |
| Flow Guardian → 作者 | **禁止** | Flow Guardian 不能强制中断作者 |
| 协调层 → 作者 | **禁止** | 协调层不能拒绝作者的有效指令 |

**唯一例外**：当作者的操作将破坏系统的一致性（如修改数据库 schema、删除审计日志）时，协调层可以拒绝执行，但必须向作者明确说明拒绝原因，并提供替代方案。这种拒绝不是"否决"，而是"边界保护"——协调层保护的是系统的技术完整性，不是叙事的正确性。

### 3.3.3 越权检测机制

NarrativeOS 内置三层越权检测机制：

**第一层：协调层实时校验**

在每个状态转换点，协调层执行实时校验：

```typescript
function validateStateTransition(
  currentState: State,
  event: HumanEvent,
  targetState: State
): ValidationResult {
  // 校验 1：事件必须来自注册的人类主权者
  if (!event.signature || !verifyHumanSignature(event.signature)) {
    return { valid: false, reason: "INVALID_SIGNATURE" };
  }
  
  // 校验 2：当前状态必须是 waiting_* 状态
  if (!currentState.startsWith("waiting_")) {
    return { valid: false, reason: "NOT_WAITING_STATE" };
  }
  
  // 校验 3：事件类型必须与当前状态匹配
  if (!isValidEventForState(currentState, event.type)) {
    return { valid: false, reason: "INVALID_EVENT_TYPE" };
  }
  
  // 校验 4：目标操作不在禁止行为清单中
  if (isForbiddenOperation(event.target, event.executor)) {
    return { valid: false, reason: "FORBIDDEN_OPERATION" };
  }
  
  // 校验 5：二次确认检查（高影响操作）
  if (requiresSecondaryConfirmation(event.target) && !event.secondaryConfirmed) {
    return { valid: false, reason: "SECONDARY_CONFIRMATION_REQUIRED" };
  }
  
  return { valid: true };
}
```

**第二层：数据库变更审计**

所有数据库变更通过审计层拦截：

```
变更请求 → 审计层拦截 → 检查人类确认锚点
    ├── 有锚点 → 允许写入 → 记录变更日志
    └── 无锚点 → 拒绝写入 → 触发越权警报
```

审计规则：
- `works` 表（作品正文）：任何 INSERT/UPDATE 必须有人类确认锚点
- `world_state` 表（世界状态）：任何 UPDATE 必须有人类确认锚点
- `audit_log` 表（审计日志）：只追加，不允许 UPDATE/DELETE
- `frozen_chapters` 表（已固化章节）：任何 UPDATE 必须经过解冻流程

**第三层：定期越权扫描**

系统定期执行越权扫描（默认每小时一次）：

1. 扫描过去一小时内的所有数据库变更
2. 验证每个变更都有对应的人类确认锚点
3. 扫描状态机日志，验证所有 `waiting_* → executing` 转移都有人类事件
4. 检查是否存在未经授权的自动行为
5. 生成越权扫描报告

### 3.3.4 审计日志

**审计日志的不可篡改性**：

审计日志是 NarrativeOS 中唯一不可修改的数据。它满足以下属性：

1. **只追加（Append-Only）**：新记录只能被追加到日志末尾，不允许修改或删除已有记录。
2. **加密链（Cryptographic Chain）**：每条记录包含前一条记录的哈希值，形成加密链。任何对历史记录的篡改都会破坏链的完整性。
3. **独立存储**：审计日志存储在独立的数据表中，与普通业务数据物理隔离。
4. **访问限制**：审计日志只能被查询，不能被任何系统组件修改。

**审计日志记录格式**：

```typescript
interface AuditRecord {
  id: string;              // 唯一标识符（UUID）
  timestamp: number;       // 时间戳（毫秒精度）
  previousHash: string;    // 前一条记录的哈希值
  currentHash: string;     // 当前记录的哈希值
  actor: "author" | "world_engine" | "studio_engine" | 
         "risk_analyzer" | "flow_guardian" | "coordinator";
  actorId: string;         // 行为者标识
  action: string;          // 行为描述
  target: string;          // 行为目标
  targetType: "content" | "world_state" | "config" | 
              "mode" | "freeze" | "thaw" | "system";
  details: object;         // 详细参数
  humanEventId?: string;   // 关联的人类事件 ID（如果有）
  result: "success" | "failure" | "blocked" | "error";
  reason?: string;         // 结果原因（失败/阻止时）
}
```

**必须记录的事件**：

| 事件类型 | 记录内容 | 强制人类锚点 |
|----------|----------|-------------|
| 正文确认 | 确认的正文内容摘要、作者 ID、时间戳 | 是 |
| 世界状态变更 | 变更前值、变更后值、变更原因、作者 ID | 是 |
| 章节固化 | 章节 ID、快照 ID、固化时间、作者 ID | 是 |
| 章节解冻 | 章节 ID、解冻原因、影响分析摘要、作者 ID | 是 |
| 提案生成 | 生成参数、提案数量、目标引擎 | 否 |
| 提案选择 | 选中的提案 ID、选择时间、作者 ID | 是 |
| 风险标记 | 风险类型、严重程度、位置、修复建议 | 否 |
| 风险处理 | 处理方式（接受/修复/忽略）、作者 ID | 是 |
| 模式切换 | 从模式、到模式、切换原因、作者 ID | 是 |
| GOD_MODE 进入/退出 | 进入/退出时间、修改摘要 | 是（退出时） |
| 混沌注入 | 混沌参数、注入结果、作者 ID | 是 |
| 配置变更 | 变更前配置、变更后配置、作者 ID | 是 |
| 紧急冻结 | 冻结原因、冻结时间、恢复时间、作者 ID | 是 |
| 越权检测 | 检测到的越权行为、处理措施、检测引擎 | N/A |
| 系统错误 | 错误类型、错误详情、恢复措施 | N/A |

## 3.4 禁止行为清单（正式版）

以下禁止行为清单是 NarrativeOS 中各引擎不可逾越的红线。任何违反都将触发越权处理流程（3.5 节）。

### 3.4.1 世界引擎禁止行为

| 编号 | 禁止行为 | 违规示例 | 检测方式 | 处罚措施 |
|------|----------|----------|----------|----------|

## 3.5 越权处理和恢复

### 3.5.1 越权检测算法

NarrativeOS 的越权检测是多层次、多触点的：

**检测层 1：协调层实时拦截（Runtime Interception）**

协调层在每个状态转换点执行实时拦截检测：

```
输入：状态转换请求 R = {from, to, event, target}
输出：ALLOW / BLOCK + reason

步骤 1：验证 event 是否有效
  - event 必须是非 null 对象
  - event.type 必须在允许的事件类型列表中
  - event.timestamp 必须在合理范围内（不超过当前时间 60 秒）
  - event.signature 必须能通过人类主权者签名验证
  → 任一失败：BLOCK, reason = "INVALID_EVENT"

步骤 2：验证 from 状态
  - from 必须是当前系统的实际状态
  - from 必须是 waiting_* 状态（或特定的非 waiting 但允许转换的状态）
  → 失败：BLOCK, reason = "INVALID_FROM_STATE"

步骤 3：验证 to 状态
  - to 必须在状态机的有效转移目标列表中
  - to 不能是终态（除非操作确实是终态操作）
  → 失败：BLOCK, reason = "INVALID_TO_STATE"

步骤 4：验证转移规则
  - (from, event.type, to) 三元组必须在转移表中存在
  → 失败：BLOCK, reason = "INVALID_TRANSITION"

步骤 5：验证目标操作
  - target.operation 不能在禁止行为清单中
  - target.executor 必须拥有执行 target.operation 的权力
  - target 是否需要二次确认（高影响检查）
  → 失败：BLOCK, reason = "FORBIDDEN_OPERATION" 或 "INSUFFICIENT_PRIVILEGE" 或 "SECONDARY_CONFIRMATION_REQUIRED"

步骤 6：记录
  - 将检测结果记录到审计日志
  - ALLOW 时记录为 "transition_approved"
  - BLOCK 时记录为 "transition_blocked"

返回：ALLOW 或 BLOCK
```

**检测层 2：数据库变更审计（Database Change Audit）**

数据库写入层拦截所有写操作：

```
输入：数据库写操作 W = {table, operation, data, context}
输出：ALLOW / BLOCK + reason

步骤 1：检查表级别的写权限
  - audit_log 表：只允许 INSERT，拒绝 UPDATE 和 DELETE
  - 若 operation 是 UPDATE/DELETE on audit_log：BLOCK, reason = "AUDIT_LOG_IMMUTABLE"

步骤 2：检查人类确认锚点
  - works, world_state, frozen_chapters 表的 INSERT/UPDATE：
    - context 中必须包含 humanEventId
    - humanEventId 必须在有效的人类事件记录中存在
    - 人类事件的状态必须是 "confirmed"
  → 失败：BLOCK, reason = "MISSING_HUMAN_ANCHOR"

步骤 3：检查固化约束
  - frozen_chapters 表的 UPDATE：
    - 必须通过解冻流程（context.thawFlow == "completed"）
  → 失败：BLOCK, reason = "FROZEN_CONTENT_PROTECTED"

步骤 4：记录
  - 允许或拒绝都记录到审计日志

返回：ALLOW 或 BLOCK
```

**检测层 3：定期完整性扫描（Periodic Integrity Scan）**

后台任务定期执行完整性扫描：

```
扫描周期：每小时一次
扫描范围：过去一小时内产生的所有记录

检查项 1：状态转移完整性
  - 获取过去一小时的全部状态转移记录
  - 验证每个 waiting_* → executing 的转移都有 humanEventId
  - 验证 humanEventId 指向的事件存在且有效
  → 发现异常：标记越权事件

检查项 2：数据库变更完整性
  - 获取 past_hour 内 works, world_state 表的所有变更
  - 验证每个变更都有对应的审计日志记录
  - 验证审计日志记录中有人类事件锚点
  → 发现异常：标记越权事件

检查项 3：禁止行为扫描
  - 扫描各引擎的输出日志
  - 检测是否存在禁止行为清单中列出的行为模式
  → 发现异常：标记越权事件

检查项 4：配置一致性
  - 验证当前系统配置与授权配置一致
  - 检测是否存在未经授权的配置变更
  → 发现异常：标记越权事件

输出：完整性扫描报告
  - 发现的问题列表
  - 每个问题的严重级别
  - 建议的处理措施
```

### 3.5.2 检测到越权后的处理流程

当任何检测层发现越权行为时，系统按以下流程处理：

**阶段 1：即时响应（毫秒级）**

```
检测触发 → 立即阻止当前操作
    ├── 若是实时拦截（检测层 1）：阻止状态转移，操作不执行
    ├── 若是数据库拦截（检测层 2）：阻止数据库写入，事务回滚
    └── 若是定期扫描（检测层 3）：标记违规记录，阻止后续相关操作
```

**阶段 2：系统冻结**

```
触发 System Freeze：
  - 暂停所有引擎的执行（已有执行中的任务允许完成但结果不生效）
  - 阻止新的状态转移请求
  - 保持当前界面状态（不丢失作者的工作）
  - 显示冻结状态指示器
```

冻结范围分级：

| 级别 | 触发条件 | 冻结范围 |
|------|----------|----------|
| **局部冻结** | 单个引擎的轻微越权 | 仅冻结越权引擎，其他引擎正常运行 |
| **全局冻结** | 协调层越权或严重越权 | 冻结全部引擎 |
| **紧急冻结** | 审计日志被篡改尝试 | 冻结全部功能，仅保留查询和恢复功能 |

**阶段 3：越权警报**

向作者发送越权警报：

```
越权警报内容：
  - 越权引擎/组件标识
  - 越权行为描述（做了什么、违反了哪条禁止行为）
  - 影响范围评估（影响了哪些数据、哪些流程）
  - 当前系统状态（冻结级别、作者的数据是否安全）
  - 建议的恢复方案
  - 作者可选操作：
    ├── 查看详细报告
    ├── 一键恢复（回滚越权操作）
    ├── 手动恢复（作者审查后选择性恢复）
    ├── 继续冻结（保持冻结状态，作者自行处理）
    └── 报告反馈（作者认为这不是越权，记录为误判）
```

**阶段 4：审计记录**

越权事件被强制记录到审计日志：

```
越权审计记录格式：
{
  type: "OVERRIDE_VIOLATION",
  severity: "critical" | "major" | "minor",
  violator: <引擎标识>,
  violatedRule: <禁止行为编号>,
  attemptedAction: <越权行为描述>,
  detectedBy: <检测层标识>,
  detectionTime: <时间戳>,
  systemState: <冻结时的系统状态>,
  affectedData: <受影响的数据范围>,
  resolution: <处理结果>,
  authorResponse: <作者的响应>
}
```

### 3.5.3 系统恢复策略

**恢复方案 A：自动回滚（Author 选择"一键恢复"）**

适用于数据库层面的越权操作：

```
1. 系统确定越权操作的时间点 T
2. 找到 T 之前最近的合法快照
3. 回滚到该快照状态
4. 重放 T 之后到冻结前的所有合法操作
5. 丢弃越权操作及其级联影响
6. 解除冻结，恢复正常运行
```

**恢复方案 B：手动恢复（Author 选择"手动恢复"）**

适用于需要作者判断的复杂场景：

```
1. 系统展示越权操作的完整影响图谱
   - 哪些数据被修改了
   - 修改前和修改后的值
   - 级联影响了哪些其他数据
2. 作者逐项决定：
   - 保留此修改（标记为"已授权的后补确认"）
   - 回滚此修改（恢复到修改前的值）
   - 修改为其他值
3. 系统执行作者的决定
4. 解除冻结，恢复正常运行
```

**恢复方案 C：误判处理（Author 选择"报告反馈"）**

当作者认为检测到的"越权"实际上是合法行为时：

```
1. 作者提交误判说明
2. 系统记录误判事件
3. 更新检测规则（如果确实是误判）
4. 解除冻结
5. 继续原操作（如果操作本身合法）
```

误判处理不会自动修改检测规则——误判记录需要经过审查后才能决定是否调整规则。这是为了防止作者因误判机制而意外降低系统的安全级别。

### 3.5.4 作者通知机制

越权事件的通知采用分级通知策略：

**通知渠道**：

| 级别 | 通知方式 | 响应要求 | 适用场景 |
|------|----------|----------|----------|
| **紧急** | 全屏模态 + 声音提示 + 系统托盘 | 必须立即响应 | 审计日志篡改尝试、全局冻结 |
| **严重** | 模态对话框 + 系统托盘 | 30 秒内响应 | 全局冻结、重要数据被修改 |
| **警告** | 非模态通知 + 系统托盘 | 下一次交互时响应 | 局部冻结、轻微越权 |
| **信息** | 状态栏提示 | 不需要立即响应 | 定期扫描发现的已阻止的越权尝试 |

**通知内容模板**：

```
[越权警报 - 级别: {severity}]

{violator} 尝试执行以下越权行为：
  {violatedRule}: {ruleDescription}

影响评估：
  - 受影响数据：{affectedData}
  - 冻结级别：{freezeLevel}
  - 您的数据状态：{dataStatus}

建议操作：
  [一键恢复]  [手动恢复]  [查看详情]  [继续冻结]
```

**通知防疲劳机制**：

1. **重复抑制**：同一引擎在 5 分钟内触发的同类越权，只发送一次通知，后续在通知中标注"还有 N 个同类事件"。
2. **升级策略**：如果同一引擎在短时间内多次越权，通知级别自动升级（警告 → 严重 → 紧急）。
3. **静默时段**：作者可以设置"免打扰时段"，在此期间越权事件只记录不通知（紧急级别的除外）。

## 3.6 权力边界的版本管理

### 3.6.1 禁止行为的版本化

禁止行为清单是系统安全策略的核心，采用版本化管理：

| 版本 | 生效日期 | 说明 |
|------|----------|------|
| v3.0.0 | 系统发布日 | 初始禁止行为清单 |

**变更规则**：
- 新增禁止行为：需要作者确认（配置变更权）
- 修改禁止行为：需要作者确认
- 删除禁止行为：需要作者确认
- 系统更新时，如果禁止行为清单有变更，系统在进入新版本前必须获得作者确认

### 3.6.2 权力清单的变更

各引擎的权力清单变更规则：

**可以增加权力的场景**（需作者确认）：
- 新增功能模块
- 作者明确授权的扩展

**必须保持的权力**（不可减少）：
- 作者的 12 项基本权力（P-01 到 P-12）
- 否决权（P-12）是不可剥夺的

**引擎间的权力边界不可由引擎自行调整**：只有作者可以修改权力边界，引擎不能自行扩展或缩小自己的权力范围。

## 3.7 合规性声明

NarrativeOS v3.0 Sovereign 的架构设计遵循以下合规性原则：

1. **人类主权优先**：所有设计决策以"人类拥有最终裁决权"为最高优先级。
2. **透明性**：系统的所有操作对作者透明，不存在不可审计的行为。
3. **可撤销性**：所有由系统执行的操作都可以被作者撤销或修改（审计日志除外，审计日志只追加）。
4. **最小权限**：每个引擎只拥有完成其职责所必需的最小权力集合。
5. **纵深防御**：越权检测采用多层次（实时拦截 + 数据库审计 + 定期扫描），确保单点故障不会导致安全失效。
6. **保守默认**：在安全与便利之间，系统默认选择安全（如高影响操作默认需要二次确认）。
