# 大纲中的伏笔规划（foreshadowing planning）说明

## 一句话解释

**在写大纲阶段，就把"哪里埋伏笔、哪里收伏笔"规划好**，防止写到后面忘了前面埋的坑。

---

## 没有伏笔规划会怎样？

### ❌ 常见问题（网文作者痛点）

```
第3章：主角发现父亲留下的神秘玉佩
    ↓
读者记住了：这个玉佩肯定有故事！
    ↓
第50章：作者忘了这回事，玉佩再也没提过
    ↓
读者骂：坑都不填！弃书！

或者反过来：

第80章：突然揭露"幕后黑手是主角父亲！"
    ↓
读者：？？？前面有任何铺垫吗？太突兀了
    ↓
读者骂：机械降神！弃书！
```

### ❌ AI 写作更容易出现的问题

- AI 有上下文长度限制（128K tokens ≈ 10万字），写到后面忘了前面的伏笔
- 没有规划时，AI 即兴埋了一个伏笔，但后续章节没有这个伏笔的记忆注入
- 结果：伏笔变"野伏笔"，有头无尾

---

## 有了伏笔规划会怎样？

### ✅ 大纲阶段的战略布局

```
大纲规划时：

第3章「父亲的遗物」  →  埋设伏笔A：神秘玉佩的来源
    │                         提示词："玉佩上的纹路似乎暗示某个古老势力"
    │
第25章「黑市惊现」   →  提及伏笔A：玉佩纹路出现在黑市拍品上
    │                         提示词："那个纹路...和玉佩一模一样！"
    │
第60章「身世之谜」   →  回收伏笔A：玉佩是上古宗门信物，主角是遗孤
    │                         提示词："玉佩亮了，宗门大阵认主"
    │
第3章「父亲的遗物」  →  同时埋设伏笔B：父亲日记中提到的"那个约定"
    │
第40章「旧友重逢」   →  提及伏笔B：旧友提到"你父亲当年答应过的事"
    │
第90章「终极真相」   →  回收伏笔B：约定关系到整个世界的存亡
```

## 在大纲表中的具体实现

### 字段用途

| 字段 | 存储 | 用途 |
|------|------|------|
| `to_plant_foreshadowings` | `fore_001`, `fore_002` | **本章计划埋设**哪些伏笔 |
| `to_resolve_foreshadowings` | `fore_001` | **本章计划回收**哪些伏笔 |
| `to_reference_foreshadowings` | `fore_003` | **本章计划提及/暗示**哪些伏笔 |

### 为什么放在 outline_items 而不是 chapters？

```
outline_items  =  蓝图（未来规划）
chapters     =  实体（现在写的）

蓝图阶段规划："第3章应该埋伏笔A"
    ↓
实际写作时：AI 生成正文，注入"伏笔A的埋设提示词"
    ↓
写完后：Memory 引擎检测 "本章是否确实埋了伏笔A？"
    ├─→ 埋了 → 更新 foreshadowings.status = 'planted'
    └─→ 没埋 → 生成风险 Proposal："大纲要求埋设伏笔A，但正文未检测到"
```

## 实际价值

### 对作者

1. **不会忘坑**：大纲阶段就规划好哪里收伏笔，系统会主动提醒
2. **布局可控**：核心伏笔的回收章节可以安排在高潮位置（如第60章、第90章）
3. **埋设密度可控**：避免"前20章埋了10个伏笔，后80章一个不收"

### 对 AI

1. **上下文精确注入**：生成第25章时，系统知道"本章需要提及伏笔A"，会在 Prompt 中注入伏笔A的信息
2. **避免幻觉**：AI 不会即兴创造新的"野伏笔"，而是严格按大纲规划执行
3. **一致性保障**：Memory 引擎会检查"大纲说埋伏笔A → 正文是否确实埋了"

### 对读者体验

- 伏笔回收在预期位置 → "卧槽，原来前面就暗示了！" → 爽
- 伏笔不回收 → "作者是不是忘了？" → 怒 → 弃书

## 如果不用，可以删掉

如果你确定不需要这个功能，设计可以简化为：

```diff
  outline_items 表：
- to_plant_foreshadowings TEXT,     -- 删除
- to_resolve_foreshadowings TEXT, -- 删除
- to_reference_foreshadowings TEXT, -- 删除
```

改为只在 `foreshadowings` 表中管理（写作过程中自然产生），不提前规划。

---

## 我的建议

**保留，但改为"可选功能"**：

- 粗纲模式：不强制要求填伏笔规划
- 细纲模式：AI 自动生成伏笔规划建议（作者可忽略）
- 不规划时：系统也不强制检测，让作者自由发挥

这样既有战略工具（想用时可用），又不增加 mandatory 负担。
