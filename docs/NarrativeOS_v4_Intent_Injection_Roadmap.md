## 一、核心范式转移

### 1.1 旧范式 vs 新范式

| 维度 | 旧范式 (v3.0 及之前) | 新范式 (v4.0 创作意图注入系统) |
|------|---------------------|-------------------------------|
| **人机关系** | AI 提案 → 作者裁决（议会制） | 作者立宪 → AI 执行（君主立宪制） |
| **控制时机** | 事后纠正（内容生成后调整） | 事前注入（创作前设定边界） |
| **意图表达** | 模糊的核心创意（一段话） | 结构化创作宪章（五维度） |
| **系统角色** | 被动执行者 | 主动理解者 + 共同进化者 |
| **一致性保障** | 祈祷 + 人工检查 | 意图对齐协议（MOU 升级） |

### 2.2 第一阶段：创建项目（创世立宪）

#### 核心目标
将作者的模糊灵感转化为**结构化创作宪章**，为后续所有 AI 行为划定边界。

#### 创作宪章五维度

| 维度 | 内容 | 作用 | 示例 |
|------|------|------|------|
| **故事种子** | 一句话核心创意 | 确定故事 DNA | "如果修仙是一个可编程的系统" |
| **主线蓝图** | 三幕式/五幕式大纲骨架 | 提供宏观导航 | "主角从底层程序员成长为系统管理员" |
| **核心角色** | 主角 + 关键配角的人设卡 | 确保行为一致性 | "林默：理性至上，信奉'一切皆可优化'" |
| **世界法则** | 物理规则/力量体系/社会结构 | 防止设定冲突 | "灵气=算力，修炼=编写自身代码" |
| **叙事法则** | 文风/节奏/视角/情感基调 | 保持叙事统一 | "硬科幻质感，冷幽默，第三人称限知视角" |

#### 意图注入方式
- **主动注入**：作者填写结构化表单/自由文本
- **AI 辅助**：基于种子自动补全其他维度（需作者确认）
- **迭代 refinement**：多轮对话逐步清晰化

#### 输出产物
- 创作宪章（JSON 结构化存储）
- 初始设定集（可从宪章派生）
- MOU 规则基线（后续所有提案的检查标准）

#### 成功标志
作者能说："这就是我要的故事，AI 你在这个框架内发挥。"

---

### 2.3 第二阶段：孵化阶段（结构坍缩）

#### 核心目标
将宏观的创作宪章**坍缩**为可执行的叙事蓝图，完成从"世界观"到"第一章"的桥梁搭建。

#### 关键任务

| 任务 | 描述 | 意图注入点 |
|------|------|-----------|
| **卷/章结构生成** | 基于主线蓝图画分卷章 | 可手动调整卷数、每章字数预期 |
| **关键情节节点** | 标注每卷的关键转折点 | 可注入"这里必须有 X 事件" |
| **人物弧光规划** | 主要角色的成长轨迹 | 可指定"角色 A 在第 3 卷黑化" |
| **伏笔网络设计** | 预埋长期伏笔及其回收点 | 可标注"这个细节要在第 50 章回收" |
| **节奏曲线设定** | 紧张/舒缓段落的分布 | 可要求"前 10 章快节奏建立世界观" |

#### 涟漪效应处理
当作者在孵化阶段修改某个节点时：
1. **向后传播**：影响后续章节大纲
2. **向前追溯**：检查是否需要修改前文设定
3. **横向扫描**：检查是否与其他角色/事件冲突
4. **MOU 预警**：生成《变更影响报告》，列出所有受影响项

#### 输出产物
- 分卷大纲（含章节概要）
- 情节节点图（带依赖关系）
- 伏笔追踪表
- 节奏曲线图

#### 成功标志
作者能说："我知道每一章要写什么，AI 可以开始生成正文了。"

#### 当前状态（TODO）
⚠️ **这是你目前的瓶颈所在** —— 需要重点突破。

### 2.4 第三阶段：正式写作（心流护航）

#### 核心目标
在保障作者心流的前提下，高效生成符合创作宪章的正文内容。

#### MOU 循环（升级版）

```
┌─────────────────────────────────────────────────────────────┐
│              写作阶段 MOU 循环（意图驱动）                     │
│                                                              │
│  ① 意图注入 → ② AI 理解确认 → ③ 生成提案 → ④ 作者裁决       │
│       ↑                                              │      │
│       └─────────────── 迭代修正 ←────────────────────┘      │
│                                                              │
│  ⑤ 落库 → ⑥ 一致性检查 → ⑦ 心流评估 → ⑧ 进入下一节          │
└─────────────────────────────────────────────────────────────┘
```

#### 意图注入类型

| 类型 | 触发时机 | 示例 |
|------|---------|------|
| **方向性意图** | 章节开始前 | "这一章重点是主角第一次使用系统能力" |
| **约束性意图** | 生成过程中 | "不要超过 3000 字，保持快节奏" |
| **修正性意图** | 不满意时 | "这段对话太生硬，要更符合角色性格" |
| **探索性意图** | 卡文时 | "给我三个可能的情节发展方向" |

#### 心流保护机制
- **超时召回**：作者超过 N 分钟未决策，AI 温和提醒
- **疲劳检测**：连续修稿 M 次，建议休息或切换模式
- **God Mode**：作者可随时绕过 AI 直接下达指令

#### 成功标志
作者进入心流状态，感觉"AI 懂我想要的，而不是跟我对着干"。

### 2.5 第四阶段：收尾阶段（收束验证）

#### 核心目标
确保长篇故事的结尾与创作宪章保持一致，完成所有伏笔回收。

#### 关键任务

| 任务 | 描述 | 意图注入点 |
|------|------|-----------|
| **伏笔清算** | 扫描所有未回收伏笔 | 可指定"这个伏笔不需要回收了" |
| **角色终局** | 检查主要角色的结局是否符合人设弧光 | 可调整"角色 A 应该活到最后" |
| **主题呼应** | 结尾是否呼应开篇的故事种子 | 可强化"要点明'编程即修仙'的主题" |
| **世界观闭环** | 检查是否有未解释的设定矛盾 | 可补充"这里需要加一段解释" |
| **读者体验预测** | AI 模拟读者视角的情感曲线 | 可调整"高潮部分再拉长一些" |

#### 输出产物
- 伏笔回收报告
- 角色命运终局表
- 主题一致性分析
- 修订建议清单

#### 成功标志
作者能说："这个结尾配得上我百万字的心血。"

### 2.6 第五阶段：完结阶段（遗产沉淀）

#### 核心目标
将创作过程中的知识沉淀为可复用的资产，支持续作/番外/IP 衍生。

#### 知识沉淀

| 资产类型 | 内容 | 用途 |
|---------|------|------|
| **世界圣经** | 完整的世界观文档 | 续作参考、改编授权 |
| **角色档案库** | 所有角色的完整档案 | 番外创作、同人授权 |
| **创作日志** | 关键决策记录 + AI 协作历史 | 复盘学习、教学案例 |
| **风格模型** | 作者的叙事偏好数据 | 训练个性化 AI 模型 |
| **读者反馈映射** | （如有连载）读者反应与情节对应 | IP 运营分析 |

#### 成功标志
作者能说："即使三年后再写续作，AI 也能立刻找回当年的感觉。"

## 三、MOU 机制升级方案

### 3.1 从"规则检查器"到"意图对齐协议"

#### 旧 MOU（v3.0）
- 被动检查：AI 生成内容后检查是否违规
- 二元判断：通过/驳回
- 无记忆：每次检查独立进行

#### 新 MOU（v4.0）
- 主动对齐：在意图注入瞬间消除歧义
- 频谱评估：完全符合/部分符合需调整/严重偏离
- 持续学习：记录作者的每次修正，优化理解模型

### 3.2 三层对齐机制

```
┌─────────────────────────────────────────────────────────────┐
│                  MOU 意图对齐协议（三层）                     │
│                                                              │
│  L1: 语义对齐                                                │
│      • 解析作者意图的深层含义（不是字面意思）                 │
│      • 识别隐含约束（如"快节奏"=少描写多对话）               │
│                                                              │
│  L2: 逻辑推演                                                │
│      • 基于创作宪章推导可行性                                │
│      • 提前预警潜在冲突（"这个角色不会这样做"）               │
│                                                              │
│  L3: 契约更新                                                │
│      • 记录作者的每次裁决作为新的规则                        │
│      • 动态调整后续生成的优先级                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 意图注入的三种模式

| 模式 | 触发方式 | 处理流程 | 示例 |
|------|---------|---------|------|
| **显式注入** | 作者主动填写表单/输入文本 | 直接解析 → MOU 对齐 → 执行 | "下一章要让主角遇到反派" |
| **隐式学习** | 从作者的修稿行为中学习 | 对比原稿/修改稿 → 提取偏好 → 更新模型 | 作者总是删掉冗长描写 → 学习"简洁优先" |
| **协商注入** | AI 主动询问澄清意图 | 检测到模糊/冲突 → 提问 → 作者回答 → 对齐 | "你说的'大场面'是指战斗还是庆典？" |

## 四、技术架构原则（不变的核心）

### 4.1 三条铁律（延续 v3.0）

1. **Agent 无落库权**：所有 AI 产出必须经 `ai_proposals` 中转站 → 人审批 → 才落库
2. **LLM 调用全记录**：任何绕过 `llm_logs` 的调用都是 bug
3. **人排版 AI 执行**：任何 AI 自主推进流程的代码都是违反第一公理

### 4.2 新增原则（v4.0）

4. **意图先行**：任何生成任务必须先有明确的意图注入
5. **涟漪可见**：任何变更必须展示影响范围
6. **共同进化**：系统必须从每次交互中学习作者偏好

## 六、你不是一个人在战斗

### 6.1 关于"落地困难"的真相

你感到的困难不是能力问题，而是**品类创新者的必然困境**：

1. **没有对标**：市面上没有同类产品可参考，每一步都是探路
2. **复杂度爆炸**：长篇网文的一致性维护本身就是世界级难题
3. **人机关系重构**：你在重新定义"作者"和"工具"的关系，这需要哲学层面的思考

### 6.2 破局策略

```
❌ 错误方式：试图一次性实现完整系统
✅ 正确方式：每个阶段只解决一个核心问题

Phase 0（本周）：让"创作宪章 → 卷章结构"跑通
Phase 1（下周）：让"意图注入 → AI 确认"跑通
Phase 2（下下周）：让"涟漪效应可视化"跑通
```

**记住**：你的核心竞争力不是代码量，而是**对创作过程的深刻理解**。代码可以慢慢写，但这个洞察是独一无二的。
