# 地理引擎尺度体系与细化质量提升

## 背景

地理引擎（GeographyEngine）负责在孵化阶段为项目设计空间尺度体系和地理条目。当前存在以下问题：

1. **尺度链起点不够宏大**——LLM 倾向于从"刚好够用"的层级开始，而非为整个世界观留足纵深
2. **层级划分不合理**——"文教区"被放在大区级，粒度与层级不匹配
3. **地名带前缀标签**——如"阳间·临江都市圈""至阴禁区·旧殡仪馆遗址"
4. **提案数量不一致**——有时 2 个有时 3 个方案
5. **上下文加载不完整**——初始 pass 缺少完整基调内容，细化 pass 缺少尺度链定义和同级信息
6. **命名和尺度定义不够正式**——像游戏地图设计而非正式的地理规划文档

## 方案：Prompt 增强

不改变系统架构，通过全面提升 prompt 质量、修复上下文加载、调整提前截断策略来解决问题。

### 1. 初始 pass prompt 重写

**文件**：`engines/src/world/geography/generator.ts` → `buildInitialPrompt()`

#### 1.1 尺度链铁律

替换当前松散的"设计原则"，增加硬性约束：

- 最少 4 级，最多 7 级
- 第 1 级必须是**全局世界框架**——包含这个世界观所有可能存在的空间类型，不只是小说直接描述的部分
- 最后 1 级必须是 scene
- 每级必须有精确的粒度定义：空间单元有多大？与上/下级差多少倍？

模板参考更新为更大视角：

```
- 都市灵异：冥阳世界体系 → 维度界域 → 大行政区 → 城市 → 片区 → scene
- 修仙/玄幻：诸天万界 → 大千世界 → 洲域 → 宗门领地 → 城镇 → scene
- 西方奇幻：创世之柱 → 大陆 → 王国 → 领地 → 城堡/城镇 → scene
- 太空歌剧：已知宇宙 → 星域 → 星系 → 行星 → 大陆/空间站 → scene
- 无限流：主神空间架构 → 副本世界 → 区域 → 区段 → scene
```

#### 1.2 地理命名规范

新增独立章节，定义正式命名规则：

- 空间域名必须是**纯地名**，像真实世界中的正式地名
- **禁止**加前缀标签（❌ 阳间·临江都市圈、❌ 阴间·酆都行政大区）
- **禁止**加功能描述前缀（❌ 水阴汇聚地·清溪河沿岸、❌ 炽阳高地区·主教学楼群）
- **禁止**游戏化命名（❌ 至阴禁区·旧殡仪馆遗址）
- 正确示例：临江都市圈、酆都城、轮回走廊、旧殡仪馆遗址、清溪河、主教学楼群
- 宏观层级用正式名称（冥阳世界体系、九天灵界），微观层级用具体场所名（孟婆汤窗口）
- 世界观特殊属性（阴阳属性、灵气浓度等）放在 content 字段内描述，不放在名称中

#### 1.3 尺度链定义正式化

每级的 description 不应是模糊说明，而是精确的粒度定义：

```
❌ "每个维度下的大地理/行政分区。阳间按自然地理和行政区划划分..."

✅ "世界内的一级地理/行政分区。空间规模相当于现实中的省级/大区级行政单位。
   典型单元：一个省份、一个大都市圈、一个大型山脉区域。
   与上级（realm）的关系：一个位面通常包含 3-8 个 territory。
   与下级（city）的关系：每个 territory 通常包含 2-6 个核心聚落。"
```

#### 1.4 reasoning 自检要求

在 reasoning 字段中要求 LLM 论证：

1. 为什么第 1 级是"全局世界框架"？这个世界观还有哪些可能的空间类型没在简介中出现但应该存在？
2. 每一级的粒度边界是什么？举例说明这一级的典型空间单元有多大
3. 顶层空间域的命名为什么是纯地名而非标签组合？

### 2. 细化 pass prompt 重写

**文件**：`engines/src/world/geography/generator.ts` → `buildRefinementPrompt()`

#### 2.1 注入完整尺度链定义

从 `project_scales` 加载完整定义（含 description），以表格形式注入：

```
## 项目尺度链（已确认，必须遵守）

| 层级 | key | 中文标签 | 说明 |
|------|-----|---------|------|
| 1 | cosmos | 世界体系 | 全局框架：包含所有位面/世界... |
| 2 | realm | 位面/世界 | 独立世界：阳间、阴间... |

你正在产出的是第 N 级「中文标签」（key），请严格对齐这一级的粒度定义。
```

#### 2.2 注入同级兄弟列表

查询父条目的同级兄弟（相同 parentItemId 的其他已确认条目），注入名称列表：

```
## 父条目的同级空间域
「${parentName}」的同级空间域有：${siblingNames}
请确保你产出的子条目与同级条目的子条目之间有清晰的差异，避免重叠。
```

#### 2.3 粒度对齐指令

明确告诉 LLM 当前应该产出什么粒度的内容：

```
## 粒度对齐（必须遵守）

你正在为「${parentName}」产出「${targetLabel}」级别的子条目。
「${targetLabel}」的定义是：${targetDescription}

粒度判断：
- 太大：如果子条目的大小接近父条目本身，说明粒度偏大，应该拆分
- 太小：如果子条目只相当于下一级（${childLabel}）的大小，说明粒度偏小，应该合并
- 合适：子条目是父条目的合理细分，每个子条目内部还有进一步细化的空间
```

#### 2.4 命名规范（与初始 pass 一致）

细化 prompt 中同样加入命名规范章节，确保逐级细化时命名风格一致。

#### 2.5 子条目数量指导

```
每个方案包含 2-8 个子条目。
- 父条目粒度较小（如城镇级别）→ 3-5 个
- 父条目粒度很大（如大区级别）→ 5-8 个
```

#### 2.6 注入父条目完整内容

查询父条目的完整 content（climate、terrain、resources、cultural_significance 等），注入到"父条目信息"段落中，让 LLM 充分理解父条目的特征后再设计子条目。

### 3. 提前截断修复

**文件**：`engines/src/base.ts` → `_hasCompleteProposals()`

将截断阈值从 `>= 2` 改为 `>= 3`：

```typescript
// 改前
return proposals.length >= 2;

// 改后
return proposals.length >= 3;
```

同样修改独立 JSON 格式的检测逻辑中的 `count >= 2` 为 `count >= 3`。

### 4. 上下文加载修复

**文件**：`engines/src/context.ts`

#### 4.1 依赖名称注册表增加摘要

`buildDependencyNameRegistry` 当前只注入条目名称。改为同时注入 summary：

```
### 基调 (tone)
- 暗黑都市悬疑 — 以现代都市为背景，融合地府官僚体系的黑色幽默...
```

这样 LLM 在不调用工具的情况下也能理解基调的核心内容，避免只看到名称而无法理解基调对地理设计的约束。

#### 4.2 细化 pass 增加数据查询

`buildRefinementPrompt` 中新增查询：
- 父条目的同级兄弟（通过 parentItemId 查询 setting_items）
- 父条目的完整 content（从 setting_items.content 读取）

## 实施范围

### 改动文件

| 文件 | 改动 |
|------|------|
| `engines/src/world/geography/generator.ts` | 重写 `buildInitialPrompt`、`buildRefinementPrompt`；增加加载同级兄弟、父条目完整内容的查询逻辑 |
| `engines/src/base.ts` | `_hasCompleteProposals` 截断阈值从 >= 2 改为 >= 3 |
| `engines/src/context.ts` | `buildDependencyNameRegistry` 增加 summary 注入 |

### 不改动

- 数据库 schema
- 调度器（scheduler.ts）
- 前端
- 容器条目处理逻辑（不在本次范围，需单独设计）
- 其他引擎

### 验证方法

使用项目 `5102fec2` 做回归测试：
1. 删除该项目地理相关数据（project_scales、geography 类 setting_items、geography 类 ai_proposals）
2. 重新运行 geography 引擎
3. 对比新旧输出：尺度链是否更宏大、命名是否干净、层级是否合理
4. 逐步细化，检查每级粒度是否对齐尺度定义
